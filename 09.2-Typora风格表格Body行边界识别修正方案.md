# 09.2-Typora风格表格Body行边界识别修正方案

关联文档：

- `09-Typora风格右键菜单交互收口与功能补齐方案.md`
- `09.1-Typora风格表格识别严格化与行列操作方案.md`

## 1. 背景

`09.1` 已经完成了源码模式表格结构化解析和首批行列操作，并且验证通过了大部分目标：

1. 表头上方普通 `A | B` 文本不会被吞进表格。
2. 列数不一致的非法表格不会被识别为可操作表格。
3. 源码模式表格右键已具备插入行、删除行、插入列、删除列、删除表格等基础能力。
4. WYSIWYG 模式没有引入脆弱 DOM 行列操作，仍保持“打开表格编辑器 + 删除表格”。

但验证时发现一个剩余边界问题：

当表格下方紧贴一行普通文本，且该普通文本中包含 `|` 并刚好拆出相同列数时，当前 `parseMarkdownTableAt()` 仍会把这行普通文本并入表格 body。

这会导致“删除表格”或行列操作误删、误改表格下方普通文本。

## 2. 问题示例

当前风险用例：

```md
| A | B |
| --- | --- |
| 1 | 2 |
普通文本 | 不是表格
```

当前错误行为：

```text
rows: [
  '| A | B |',
  '| --- | --- |',
  '| 1 | 2 |',
  '普通文本 | 不是表格'
]
```

期望行为：

```text
rows: [
  '| A | B |',
  '| --- | --- |',
  '| 1 | 2 |'
]
```

也就是说，`普通文本 | 不是表格` 不能被纳入表格范围。

## 3. 修复目标

本次只修一个边界：

```text
表格 body 向下扩展时，不再仅凭“包含 | 且列数一致”判断为表格行。
```

完成后应保证：

1. 标准 GFM 表格仍可识别。
2. 表格下方普通 `|` 文本不会被纳入表格。
3. 删除表格不会误删表格下方普通文本。
4. 插入行列、删除行列不会改写表格下方普通文本。
5. 不扩大本轮范围，不做对齐子菜单，不做 WYSIWYG 行列操作。

## 4. 建议修复策略

### 4.1 首版采用边界管道约束

为了降低误判，首版建议要求源码模式可编辑表格必须采用边界管道写法。

合法 body 行：

```md
| 1 | 2 |
```

不纳入 body 的普通文本：

```md
普通文本 | 不是表格
```

建议新增判断函数：

```ts
function isBoundaryPipeRow(line: string): boolean {
  const t = line.trim()
  return t.startsWith('|') && t.endsWith('|')
}
```

然后在 body 向下扩展时增加限制：

```ts
if (!isBoundaryPipeRow(lt)) break
```

### 4.2 表头和分隔行也建议统一要求边界管道

`09.1` 文档曾提到第一版建议优先支持带边界 `|` 的标准写法。因此本次可以把严格识别范围固定为边界管道表格。

建议规则：

1. 表头行必须是边界管道行。
2. 分隔行必须是边界管道行。
3. body 行必须是边界管道行。
4. 三者列数必须一致。

这样可明确排除以下无边界写法：

```md
A | B
--- | ---
1 | 2
```

这类无边界表格可以作为后续增强，不应影响当前“安全删除和行列操作”的目标。

### 4.3 弱匹配仍可保留

对于不满足严格识别的 `|` 行，可以继续保留弱匹配策略：

1. 如果当前行以 `|` 开头，但无法解析为合法表格，可以显示“打开表格编辑器”。
2. 不显示“删除表格”。
3. 不显示行列操作。

普通文本 `普通文本 | 不是表格` 不以 `|` 开头，因此不应进入表格菜单。

## 6. 具体实施步骤

### 6.1 新增边界管道判断

在 `src/utils/markdownTable.ts` 中新增：

```ts
function isBoundaryPipeRow(line: string): boolean {
  const t = line.trim()
  return t.startsWith('|') && t.endsWith('|')
}
```

### 6.2 收紧分隔行识别

`isSeparatorRow()` 可以继续负责判断单元格内容是否都是分隔符，但调用它之前或内部应确保该行是边界管道行。

推荐：

```ts
export function isSeparatorRow(line: string): boolean {
  if (!isBoundaryPipeRow(line)) return false
  const cells = splitPipeRow(line)
  return cells.length > 0 && cells.every(isSeparatorCell)
}
```

### 6.3 收紧候选表头

在 `parseMarkdownTableAt()` 中校验候选表头：

```ts
if (!isBoundaryPipeRow(headerText)) continue
```

### 6.4 收紧 body 扩展

在 body 向下扫描时：

```ts
if (!isBoundaryPipeRow(lt)) break
```

放置位置建议在 `containsPipe(lt)` 之后、`splitPipeRow(lt)` 之前。

### 6.5 保持弱匹配策略不变

`editorContextMenu.ts` 中弱匹配逻辑可以继续保持：

```ts
if (lt.includes('|') && /^\s*\|/.test(ltPipe)) return { kind: 'table' }
```

这样不合法但以 `|` 开头的行仍可显示“打开表格编辑器”，但不会出现删除表格和行列操作。

## 7. 验收用例

### 7.1 标准表格仍可识别

输入：

```md
| A | B |
| --- | --- |
| 1 | 2 |
```

预期：

1. `parseMarkdownTableAt()` 返回表格。
2. `rows.length === 3`。
3. 显示删除表格和行列操作。

### 7.2 表格下方普通文本不被纳入

输入：

```md
| A | B |
| --- | --- |
| 1 | 2 |
普通文本 | 不是表格
```

预期：

1. 右键前三行任意位置，解析出的表格只包含前三行。
2. `普通文本 | 不是表格` 不在 `rows` 中。
3. 删除表格后，普通文本保留。

### 7.3 普通 `|` 文本不识别为表格

输入：

```md
普通文本 | 不是表格
```

预期：

1. `parseMarkdownTableAt()` 返回 `null`。
2. 不显示删除表格。
3. 不显示行列操作。

### 7.4 无边界管道表格暂不进入严格操作

输入：

```md
A | B
--- | ---
1 | 2
```

预期：

1. `parseMarkdownTableAt()` 返回 `null`。
2. 不显示删除表格。
3. 不显示行列操作。

说明：这是当前阶段有意保守的行为，后续如果要支持无边界 GFM 表格，应单独设计和测试。

### 7.5 非法列数仍返回 null

输入：

```md
| A | B |
| --- |
| 1 | 2 |
```

预期：

1. `parseMarkdownTableAt()` 返回 `null`。
2. 不显示删除表格。
3. 不显示行列操作。

## 8. 回归验证命令

建议执行：

```bash
npm run type-check
npx vite build
```

建议补充一次临时脚本或手动样例验证：

```ts
parseMarkdownTableAt(Text.of(source.split('\n')), pos)
```

重点确认：

1. 标准表格可识别。
2. 表格下方普通 `|` 文本不被纳入。
3. 普通 `|` 文本返回 `null`。
4. 无边界表格暂不进入严格操作。

## 9. 非目标

本次不做：

1. 对齐方式子菜单。
2. WYSIWYG 表格行列操作。
3. 无边界 GFM 表格支持。
4. 转义 `\|` 的复杂解析。
5. HTML 表格解析。
6. 表格格式化美化。

## 10. 完成标准

本方案完成后，应满足：

```text
严格表格解析只接受边界管道表格，表格下方普通包含 | 的文本不会被纳入表格范围。
```

只有该边界修复通过后，才建议继续推进表格对齐子菜单或更高级的表格能力。
