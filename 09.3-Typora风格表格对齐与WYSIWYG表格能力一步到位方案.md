# 09.3-Typora风格表格对齐与WYSIWYG表格能力一步到位方案

关联文档：

- `09-Typora风格右键菜单交互收口与功能补齐方案.md`
- `09.1-Typora风格表格识别严格化与行列操作方案.md`
- `09.2-Typora风格表格Body行边界识别修正方案.md`

## 1. 背景

当前表格右键菜单已经完成以下基础：

1. 源码模式严格识别边界管道表格。
2. 源码模式支持删除表格、插入行、删除行、插入列、删除列。
3. 表格下方普通 `|` 文本不会再被误吞进表格范围。
4. WYSIWYG 模式保持稳定能力：打开表格编辑器、删除表格。
5. WYSIWYG 没有使用脆弱 DOM 方式做行列操作。

剩余需要一次性收口的内容：

1. 源码模式表格对齐子菜单。
2. WYSIWYG 模式表格行列操作。
3. WYSIWYG 模式表格对齐能力。
4. 源码模式和 WYSIWYG 模式的表格右键体验尽量一致。

本方案目标是“一步到位”：不再只做提示或禁用，不再保留半成品入口。源码和 WYSIWYG 都应具备完整的 Typora 风格表格右键基础能力。

## 2. 总目标

完成后，表格右键菜单应支持：

```text
打开表格编辑器
--------
插入行到上方
插入行到下方
插入列到左侧
插入列到右侧
--------
删除当前行
删除当前列
删除表格
--------
对齐方式 >
  默认对齐
  左对齐
  居中对齐
  右对齐
```

源码模式和 WYSIWYG 模式都应遵循：

```text
右键哪一个表格单元格，就操作该单元格所在行、列和表格。
```

## 3. 非目标

本轮不做：

1. 合并单元格。
2. 拆分单元格。
3. 拖拽调整列宽。
4. HTML 表格解析。
5. 无边界管道表格的严格编辑。
6. 复杂转义 `\|` 的完整解析。
7. 表格主题样式和视觉美化。

## 4. 源码模式：表格对齐子菜单

### 4.1 当前基础

当前 `src/utils/markdownTable.ts` 已有：

1. `MarkdownTableBlock`
2. `MarkdownTableRow`
3. `MarkdownTableCellPosition`
4. `parseMarkdownTableAt()`
5. `serializeMarkdownTable()`
6. `insertRowAbove()`
7. `insertRowBelow()`
8. `deleteBodyRow()`
9. `insertColumnLeft()`
10. `insertColumnRight()`
11. `deleteColumn()`

但当前序列化会统一输出：

```md
| A | B |
| --- | --- |
```

还没有保留或设置对齐信息。

### 4.2 数据模型扩展

在 `src/utils/markdownTable.ts` 中新增对齐类型：

```ts
export type MarkdownTableAlignment = 'default' | 'left' | 'center' | 'right'
```

扩展 `MarkdownTableBlock`：

```ts
export interface MarkdownTableBlock {
  from: number
  to: number
  startLine: number
  endLine: number
  columnCount: number
  rows: MarkdownTableRow[]
  currentCell?: MarkdownTableCellPosition
  trailingNewline: boolean
  alignments: MarkdownTableAlignment[]
}
```

### 4.3 分隔行对齐解析

新增：

```ts
function parseSeparatorAlignment(cell: string): MarkdownTableAlignment
```

规则：

```text
---    -> default
:---   -> left
:---:  -> center
---:   -> right
```

注意：

1. `:---` 和 `---` 在 Markdown 渲染中都可能显示为左对齐，但语义上应区分 `default` 和 `left`。
2. `default` 表示不显式写冒号。
3. `left` 表示显式 `:---`。

### 4.4 分隔行序列化

新增：

```ts
function alignmentToSeparatorCell(alignment: MarkdownTableAlignment): string
```

规则：

```text
default -> ---
left    -> :---
center  -> :---:
right   -> ---:
```

`serializeMarkdownTable()` 应根据 `block.alignments` 输出分隔行，而不是直接序列化 `separator` 行原始 cells。

### 4.5 列操作维护对齐数组

插入列：

1. 插入空表头单元格。
2. 插入默认分隔单元格。
3. 插入空 body 单元格。
4. `alignments` 同步插入 `'default'`。

删除列：

1. 删除所有行对应单元格。
2. `alignments` 同步删除对应列。
3. 至少保留一列。

### 4.6 新增设置列对齐函数

新增：

```ts
export function setColumnAlignment(
  block: MarkdownTableBlock,
  colIdx: number,
  alignment: MarkdownTableAlignment,
): MarkdownTableBlock
```

行为：

1. 克隆表格块。
2. 只修改 `alignments[colIdx]`。
3. 不改变任何单元格内容。
4. 不改变行数和列数。

### 4.7 源码右键菜单接入

在 `src/utils/editorContextMenu.ts` 的源码表格菜单中增加子菜单：

```text
对齐方式 >
  默认对齐
  左对齐
  居中对齐
  右对齐
```

每个菜单项调用：

```ts
replaceTable(setColumnAlignment(tbl, colIdx, alignment))
```

如果当前 `ContextMenuEntry` 尚不支持选中态，可以使用标签文案表达当前状态：

```text
默认对齐（当前）
左对齐
居中对齐
右对齐
```

如果希望 UI 更清晰，可以给 `ContextMenuEntry` 增加：

```ts
checked?: boolean
```

并在 `ContextMenuPanel.vue` 中显示选中标记。但这不是必须项。

## 5. WYSIWYG：表格操作总策略

### 5.1 禁止 DOM 直接改表格

WYSIWYG 表格行列操作必须通过 ProseMirror transaction 或 Milkdown 命令完成。

禁止：

1. 直接操作 `<table>` DOM。
2. 直接操作 `<tr>` DOM。
3. 直接操作 `<td>` DOM。
4. 用 `innerHTML` 拼表格。

原因：

1. DOM 改动不会稳定同步到编辑器 state。
2. 容易破坏 Milkdown/ProseMirror selection。
3. 容易造成保存内容和界面内容不一致。

### 5.2 建议新增 ProseMirror 表格工具模块

新增文件：

- `src/utils/proseMirrorTable.ts`

该文件只做 ProseMirror 表格结构定位和 transaction 辅助，不依赖 Vue。

建议类型：

```ts
import type { EditorState, Transaction } from '@milkdown/prose/state'
import type { Node as ProseNode, NodeType } from '@milkdown/prose/model'

export type PmTableAlignment = 'default' | 'left' | 'center' | 'right'

export interface PmTableCellInfo {
  rowIndex: number
  columnIndex: number
  from: number
  to: number
  node: ProseNode
}

export interface PmTableRowInfo {
  rowIndex: number
  from: number
  to: number
  node: ProseNode
  cells: PmTableCellInfo[]
}

export interface PmTableTarget {
  tableFrom: number
  tableTo: number
  tableNode: ProseNode
  rowIndex: number
  columnIndex: number
  rows: PmTableRowInfo[]
  columnCount: number
}
```

### 5.3 表格定位

新增：

```ts
export function resolvePmTableTarget(state: EditorState, pos: number): PmTableTarget | null
```

定位规则：

1. 从 `pos` 向上查找 `table` 节点。
2. 读取 table 内部 row 节点。
3. row 节点名称兼容：
   - `table_row`
   - 包含 `row` 的表格行节点名
4. cell 节点名称兼容：
   - `table_cell`
   - `table_header`
   - 包含 `cell` 或 `header` 的表格单元格节点名
5. 根据 `pos` 判断当前 rowIndex 和 columnIndex。

如果无法稳定定位当前单元格，应返回 `null`，不显示行列操作。

### 5.4 空单元格创建

新增：

```ts
function createEmptyCell(state: EditorState, cellType: NodeType): ProseNode
```

规则：

1. 优先创建包含空 `paragraph` 的 cell。
2. 如果 schema 不允许 paragraph，则创建空 cell。
3. cell type 尽量复用同一行相邻 cell 的类型。
4. 表头行插入列时，可复用该行当前 cell 类型。

### 5.5 表格节点替换策略

为了降低直接位置计算复杂度，WYSIWYG 行列操作建议采用“重建 table node 并替换整个 table”的策略：

1. 根据 `PmTableTarget.tableNode` 克隆 rows。
2. 对 rows/cells 做结构修改。
3. 创建新的 table node。
4. 用 transaction 替换 `[tableFrom, tableTo]`。

优点：

1. 不需要逐个计算 cell 插入位置。
2. 行列一致性更容易保证。
3. 逻辑和源码模式的“序列化整表替换”相似。

注意：

1. 必须保留每个现有 cell 的内容。
2. 必须尽量保留 row/cell attrs。
3. 必须保留 table attrs。

## 6. WYSIWYG：行操作

### 6.1 插入行到上方

新增函数：

```ts
export function insertPmTableRowAbove(state: EditorState, target: PmTableTarget): Transaction | null
```

规则：

1. 新行列数等于 `target.columnCount`。
2. 新行 cell 类型参考当前行的 cell 类型。
3. 新行插入到当前行上方。
4. 如果当前行是第一行，允许在第一行下方还是上方需要产品明确。

建议本项目策略：

```text
第一行视为表头行，不在其上方插入数据行。
如果右键第一行并选择“插入行到上方”，则插入到第一条 body 行上方。
```

这样与源码模式“表头和分隔行受保护”的策略一致。

### 6.2 插入行到下方

新增：

```ts
export function insertPmTableRowBelow(state: EditorState, target: PmTableTarget): Transaction | null
```

规则：

1. 如果当前行是表头行，则插入到第一条 body 行位置。
2. 如果当前行是 body 行，则插入到当前行下方。
3. 新行为空单元格。

### 6.3 删除当前行

新增：

```ts
export function deletePmTableRow(state: EditorState, target: PmTableTarget): Transaction | null
```

规则：

1. 第一行视为表头行，不允许删除。
2. 至少保留一行 body；如果只剩表头行和一条 body 行，删除 body 后允许只保留表头行还是禁用，需要统一。

建议本项目策略：

```text
不允许删除第一行。
允许删除最后一条 body 行，删除后保留只有表头的表格。
```

如果实际 schema 不允许空 body，也应禁用删除最后一条 body。

### 6.4 菜单禁用规则

1. 当前行是第一行：禁用“删除当前行”。
2. 无法定位 `rowIndex`：隐藏所有行操作。
3. 无法创建合法新行：禁用插入行操作并提示。

## 7. WYSIWYG：列操作

### 7.1 插入列到左侧

新增：

```ts
export function insertPmTableColumnLeft(state: EditorState, target: PmTableTarget): Transaction | null
```

规则：

1. 每一行都插入一个空 cell。
2. 新 cell 类型参考该行当前列 cell。
3. 插入位置是当前列左侧。

### 7.2 插入列到右侧

新增：

```ts
export function insertPmTableColumnRight(state: EditorState, target: PmTableTarget): Transaction | null
```

规则：

1. 每一行都插入一个空 cell。
2. 插入位置是当前列右侧。

### 7.3 删除当前列

新增：

```ts
export function deletePmTableColumn(state: EditorState, target: PmTableTarget): Transaction | null
```

规则：

1. 至少保留一列。
2. 每一行同步删除当前列。
3. 如果某行缺少该列，返回 `null`，不做部分修改。

### 7.4 菜单禁用规则

1. `columnCount <= 1` 时禁用“删除当前列”。
2. 无法定位 `columnIndex` 时隐藏列操作。
3. 表格行列数量不一致时隐藏行列操作，只保留打开表格编辑器和删除表格。

## 8. WYSIWYG：对齐方式

### 8.1 优先方案：使用 schema attrs

先检查 Milkdown/GFM table cell schema 是否支持对齐 attrs。

可能字段：

1. `align`
2. `alignment`
3. `textAlign`

实现前应在运行时或源码中确认 table cell attrs。

新增：

```ts
export function setPmTableColumnAlignment(
  state: EditorState,
  target: PmTableTarget,
  alignment: PmTableAlignment,
): Transaction | null
```

规则：

1. 对当前列所有 cell/header cell 设置对齐 attr。
2. 不改动 cell 内容。
3. 不改动行列数量。
4. 如果 schema 不支持对齐 attrs，则使用备用方案。

### 8.2 备用方案：Markdown 文本级对齐

如果 ProseMirror schema 不支持对齐 attrs，不能强行用 DOM style。

备用方案：

1. WYSIWYG 对齐菜单触发时，显示提示：
   ```text
   当前 WYSIWYG 表格结构暂不支持直接设置列对齐，请切换源码模式或使用表格编辑器。
   ```
2. 同时保持源码模式对齐完整可用。

但用户这次要求“一步到位”，因此实施时应优先调查 schema attrs，尽量完成 WYSIWYG 对齐。

只有确认 schema 无法保存对齐信息时，才允许采用提示方案。不能使用 DOM style 假装完成。

### 8.3 对齐菜单文案

WYSIWYG 和源码模式保持同一菜单：

```text
对齐方式 >
  默认对齐
  左对齐
  居中对齐
  右对齐
```

如果无法读取当前对齐状态，可以不显示“当前”标记，但点击后必须产生可保存的文档变化，或者明确提示不支持。

## 11. 验收标准

### 11.1 源码模式对齐

输入：

```md
| A | B | C |
| --- | --- | --- |
| 1 | 2 | 3 |
```

右键第二列，选择“居中对齐”。

预期：

```md
| A | B | C |
| --- | :---: | --- |
| 1 | 2 | 3 |
```

右键第三列，选择“右对齐”。

预期：

```md
| A | B | C |
| --- | :---: | ---: |
| 1 | 2 | 3 |
```

右键第二列，选择“默认对齐”。

预期：

```md
| A | B | C |
| --- | --- | ---: |
| 1 | 2 | 3 |
```

### 11.2 源码模式列操作保留对齐

输入：

```md
| A | B |
| :--- | ---: |
| 1 | 2 |
```

在 B 左侧插入列。

预期：

```md
| A |  | B |
| :--- | --- | ---: |
| 1 |  | 2 |
```

删除 A 列。

预期：

```md
| B |
| ---: |
| 2 |
```

### 11.3 WYSIWYG 行操作

1. 右键 body 行，插入上方行，新增空行在当前行上方。
2. 右键 body 行，插入下方行，新增空行在当前行下方。
3. 右键第一行，删除当前行禁用。
4. 右键 body 行，删除当前行，只删除该行。
5. 删除行后表格仍可继续编辑和保存。

### 11.4 WYSIWYG 列操作

1. 右键某列，插入左侧列，所有行增加一列。
2. 右键某列，插入右侧列，所有行增加一列。
3. 删除当前列，所有行同步删除该列。
4. 只剩一列时，删除当前列禁用。
5. 操作后文档保存再打开，表格结构仍正确。

### 11.5 WYSIWYG 对齐

如果 schema 支持对齐 attrs：

1. 右键当前列，选择居中对齐。
2. 预览或保存后的 Markdown 能体现 `:---:`。
3. 再选择右对齐，保存后能体现 `---:`。
4. 再选择默认对齐，保存后恢复 `---`。

如果 schema 不支持对齐 attrs：

1. 菜单项应明确提示不支持。
2. 不允许用 DOM style 临时显示对齐。
3. 不允许让用户误以为已经保存。

3. WYSIWYG 设置对齐后切到源码，分隔行体现对应对齐。
4. 源码插入列后切到 WYSIWYG，表格列数正确。

## 14. 风险与兜底

### 14.1 WYSIWYG schema 差异

Milkdown/GFM table 的内部 schema 可能与预期不同。

兜底要求：

1. 行列操作必须先通过 schema 创建合法 node。
2. 如果无法创建合法 row/cell，禁用对应菜单并提示。
3. 不能使用 DOM 操作绕过 schema。

### 14.2 对齐 attrs 不可保存

如果 ProseMirror table schema 不支持对齐 attrs，WYSIWYG 对齐不能假完成。

兜底：

1. 源码模式完整支持对齐。
2. WYSIWYG 对齐菜单提示“不支持直接设置，请切换源码模式或使用表格编辑器”。
3. 后续若引入支持对齐 attrs 的表格 schema，再开启 WYSIWYG 对齐。

### 14.3 整表替换导致 selection 丢失

行列操作采用整表替换后，selection 可能回到表格前后。

建议：

1. 操作后将 selection 放到新插入行/列附近。
2. 如果定位复杂，至少保持编辑器 focus。
3. 不因 selection 不完美阻塞结构正确性。

## 15. 完成标准

本方案完成后，应达到：

1. 源码模式表格支持行列操作和对齐方式。
2. 源码模式对齐信息在插入列、删除列后仍保持正确。
3. WYSIWYG 模式不再只支持“打开表格编辑器 + 删除表格”，而是支持基础行列操作。
4. WYSIWYG 表格操作不使用 DOM 直接修改。
5. 源码和 WYSIWYG 的表格菜单结构一致。
6. 不支持的 WYSIWYG 对齐场景必须明确提示，不能伪装成功。

最终验收标准：

```text
表格右键菜单在源码模式和 WYSIWYG 模式中都能可靠完成基础行列编辑；源码模式完整支持列对齐，WYSIWYG 对齐要么可保存，要么明确提示不可用。
```
