# 04.1-快捷键体系剩余收口方案

关联方案：`04-快捷键体系统一与文档同步方案.md`  
当前状态：运行时快捷键注册表、菜单/欢迎页/状态栏/README 的主要冲突已修正，`npm run type-check` 已通过。

## 1. 剩余问题

本轮验证后，仍有两个点没有完全达到“一劳永逸”的标准：

1. README 快捷键表仍是手动维护，没有自动同步脚本。
2. `Ctrl+Tab` 仍是帮助弹窗和欢迎页里手动追加，没有作为特殊快捷键纳入注册表。

这两个问题短期不影响当前快捷键行为，但会影响长期维护。后续如果新增或修改快捷键，README 和特殊快捷键说明仍可能再次漂移。

## 2. 目标

本方案目标是把快捷键体系最后两个维护风险收口：

1. README 中的快捷键表由 `shortcutRegistry.ts` 自动生成。
2. `Ctrl+Tab` 作为特殊快捷键纳入注册表描述，帮助弹窗和欢迎页不再手动追加。
3. 开发流程中形成明确规则：改快捷键必须改注册表，README 由脚本生成。

## 3. 剩余点一：README 自动同步

### 3.1 当前问题

当前 `README.md` 已经手动改正确：

- 大纲：`Ctrl+Shift+\`
- 字号：说明为“视图菜单操作，无默认全局快捷键”

但它仍然是手动内容，没有以下机制：

- `<!-- shortcuts:start -->`
- `<!-- shortcuts:end -->`
- `scripts/update-readme-shortcuts.ts`
- `npm run docs:shortcuts`

因此后续如果修改 `shortcutRegistry.ts`，README 不会自动更新。

### 3.2 解决思路

把 README 快捷键表改成“受管区块”，由脚本根据 `src/utils/shortcutRegistry.ts` 生成。

README 中只保留固定标记：

```md
<!-- shortcuts:start -->
| 快捷键 | 作用 |
|--------|------|
| Ctrl+N | 新建文件 |
<!-- shortcuts:end -->
```

标记之间的内容由脚本覆盖，不再手写。

### 3.3 脚本设计

新增文件：

```text
scripts/update-readme-shortcuts.ts
```

脚本职责：

1. 读取快捷键注册表。
2. 生成 Markdown 表格。
3. 替换 README 中 `shortcuts:start/end` 之间的内容。
4. 保留表格下方的额外说明，例如字号无默认快捷键。

### 3.4 推荐生成规则

README 快捷键表不需要列出所有编辑命令，可以面向普通用户展示常用项。

建议使用 `visibleInHelp` 或新增 `visibleInReadme` 字段。

推荐新增字段：

```ts
visibleInReadme?: boolean
```

原因：

- `visibleInHelp` 适合完整帮助弹窗。
- `visibleInWelcome` 适合欢迎页，只展示少量入口。
- `visibleInReadme` 适合用户文档，比帮助弹窗更精简。

如果暂时不新增字段，也可以用一个固定 ID 白名单生成 README：

```ts
const README_SHORTCUT_IDS = [
  'file-new',
  'file-open',
  'file-save',
  'file-save-as',
  'file-close-tab',
  'view-cycle-mode',
  'edit-find',
  'edit-replace',
  'view-toggle-toc',
  'view-sidebar',
  'view-focus',
  'table-open',
  'tabs-cycle',
  'theme-cycle',
]
```

长期建议使用 `visibleInReadme`。

### 3.5 package.json 脚本

需要新增：

```json
{
  "scripts": {
    "docs:shortcuts": "tsx scripts/update-readme-shortcuts.ts"
  }
}
```

如果项目暂时没有 `tsx`，有两个选择：

1. 增加开发依赖 `tsx`。
2. 用纯 Node 脚本维护一份可导入的 JSON 快捷键数据。

推荐使用 `tsx`，因为当前项目已使用 TypeScript，脚本可以直接复用 `shortcutRegistry.ts`。

### 3.6 README 更新流程

后续修改快捷键时流程固定为：

1. 修改 `src/utils/shortcutRegistry.ts`。
2. 执行：

```bash
npm run docs:shortcuts
```

3. 执行：

```bash
npm run type-check
```

4. 检查 README diff。

### 3.7 验收标准

完成后应满足：

- `README.md` 包含 `<!-- shortcuts:start -->` 和 `<!-- shortcuts:end -->`。
- 快捷键表由脚本生成，不再手动维护。
- `package.json` 包含 `docs:shortcuts`。
- 修改注册表后运行脚本，README 快捷键表会同步变化。
- 字号无默认快捷键的说明保留在自动生成区块之外，或由脚本固定生成。

## 4. 剩余点二：Ctrl+Tab 纳入注册表

### 4.1 当前问题

`Ctrl+Tab` 不是普通 renderer keydown 快捷键。

当前流程是：

1. Electron 主进程在 `electron/main/index.ts` 中监听 `before-input-event`。
2. 捕获 `Ctrl+Tab` 后发送 `shortcut:ctrl-tab` 到 renderer。
3. `App.vue` 中 `setupCtrlTab()` 监听 `window.electronAPI.onCtrlTab()`，切换标签。

因此它没有放在 `SHORTCUTS` 中，而是在：

- `shortcutsForHelpDialog()`
- `shortcutsForWelcome()`

里手动追加。

这会破坏“快捷键说明只来自注册表”的原则。

### 4.2 解决思路

把 `Ctrl+Tab` 作为一种“外部派发快捷键”纳入注册表，只参与展示和冲突记录，不参与普通 `keydown` 分发。

### 4.3 类型设计

建议扩展 `ShortcutAction`：

```ts
export type ShortcutAction =
  | { kind: 'command'; command: AppCommandDetail }
  | { kind: 'internal'; id: InternalShortcutId }
  | { kind: 'external'; id: ExternalShortcutId }
```

新增：

```ts
export type ExternalShortcutId =
  | 'cycle-tabs'
```

或者更简单：

```ts
export interface ShortcutDefinition {
  id: string
  label: string
  display: string
  chord: string
  scope: ShortcutScope
  action: ShortcutAction
  dispatchMode?: 'renderer-keydown' | 'main-process'
  visibleInHelp?: boolean
  visibleInWelcome?: boolean
  visibleInReadme?: boolean
}
```

`Ctrl+Tab` 定义为：

```ts
{
  id: 'tabs-cycle',
  label: '切换标签页',
  display: 'Ctrl+Tab',
  chord: 'ctrl+tab',
  scope: 'global',
  action: { kind: 'external', id: 'cycle-tabs' },
  dispatchMode: 'main-process',
  visibleInHelp: true,
  visibleInWelcome: true,
  visibleInReadme: true,
}
```

### 4.4 normalizeKeyboardChord 支持 Tab

虽然 `Ctrl+Tab` 不走普通 renderer keydown，但注册表冲突检测仍需要标准 chord。

建议 `normalizeKeyboardChord()` 支持：

```ts
case 'Tab':
  key = 'tab'
  break
```

即使它不参与普通查找，也保持 chord 规范完整。

### 4.5 findShortcutDefinitionByChord 的处理

如果 `tabs-cycle` 放入 `SHORTCUTS`，但普通 renderer keydown 收到 `Ctrl+Tab` 时不应重复处理。

有两种做法：

#### 做法 A：查找时跳过非 renderer-keydown

```ts
export function findShortcutDefinitionByChord(chord: string | null): ShortcutDefinition | null {
  if (!chord) return null
  return SHORTCUTS.find(s =>
    s.chord === chord &&
    (s.dispatchMode ?? 'renderer-keydown') === 'renderer-keydown'
  ) ?? null
}
```

#### 做法 B：App.vue 分发时忽略 external

```ts
if (def.action.kind === 'external') return
```

推荐做法 A，因为查找结果本身就代表“可由 renderer keydown 分发”。

### 4.6 shortcutsForHelpDialog / shortcutsForWelcome 改造

当前这两个函数手动追加 `Ctrl+Tab`。

完成注册表纳入后，应删除手动追加逻辑：

```ts
const head = [
  { keys: 'Ctrl+Tab', label: '切换标签页（Windows 下由主进程拦截）' },
  // ...
]
```

改成从 `SHORTCUTS` 统一生成。

如果仍想保留“Windows 下由主进程拦截”的说明，可以在注册表中增加 `note` 字段：

```ts
note?: string
```

展示时：

```ts
label: s.note ? `${s.label}（${s.note}）` : s.label
```

### 4.7 README 展示

README 快捷键表应由注册表生成，因此 `Ctrl+Tab` 也来自注册表，而不是 README 手写。

最终 README 表中展示：

```md
| Ctrl+Tab | 切换标签页 |
```

可选补充：

```md
注：Windows 下 `Ctrl+Tab` 由主进程拦截以保证标签切换一致。
```

### 4.8 验收标准

完成后应满足：

- `Ctrl+Tab` 在 `SHORTCUTS` 中有定义。
- `shortcutsForHelpDialog()` 不再手动追加 `Ctrl+Tab`。
- `shortcutsForWelcome()` 不再手动追加 `Ctrl+Tab`。
- `Ctrl+Tab` 不会被 renderer 普通 keydown 重复分发。
- 帮助弹窗、欢迎页、README 中的 `Ctrl+Tab` 都来自注册表。

## 5. 推荐最终实现顺序

### 第一阶段：纳入 Ctrl+Tab

1. 扩展 `ShortcutAction` 或增加 `dispatchMode`。
2. 在 `SHORTCUTS` 中添加 `tabs-cycle`。
3. 修改 `findShortcutDefinitionByChord()`，跳过 `dispatchMode: 'main-process'`。
4. 删除 `shortcutsForHelpDialog()` 和 `shortcutsForWelcome()` 里的手动追加。
5. 跑 `npm run type-check`。

### 第二阶段：README 自动同步

1. 为 `SHORTCUTS` 增加 `visibleInReadme`，或定义 README 白名单。
2. 在 README 中加入 `shortcuts:start/end` 标记。
3. 新增 `scripts/update-readme-shortcuts.ts`。
4. 新增 `npm run docs:shortcuts`。
5. 运行脚本更新 README。
6. 跑 `npm run type-check`。

### 第三阶段：流程固化

1. 在 `04-快捷键体系统一与文档同步方案.md` 或开发说明中补充流程：

```text
修改快捷键时，必须修改 shortcutRegistry.ts，并运行 npm run docs:shortcuts。
```

2. 后续如果增加测试，将 `validateShortcutRegistry()` 和 README 生成脚本纳入 CI。

## 6. 风险与注意事项

### 6.1 不要让 Ctrl+Tab 双触发

`Ctrl+Tab` 已经由主进程处理，不要再让 renderer 普通 keydown 同时处理。

注册表中应明确它是：

```ts
dispatchMode: 'main-process'
```

### 6.2 README 脚本不要解析 Vue 文件

README 应从 `shortcutRegistry.ts` 生成，不应该反向解析 `App.vue` 或菜单模板。

### 6.3 自动生成区块外允许人工说明

例如字号没有默认快捷键，可以放在快捷键表下面：

```md
界面字号请在视图菜单中操作，未绑定默认全局快捷键。
```

这类说明可以手写，也可以由脚本固定写入，但不要混在受管快捷键表里。

## 7. 最终验收标准

完成本方案后，快捷键体系应满足：

1. `SHORTCUTS` 包含所有用户可见快捷键，包括 `Ctrl+Tab`。
2. 普通 renderer 快捷键和主进程快捷键都有明确 `dispatchMode`。
3. 帮助弹窗、欢迎页、菜单快捷键和 README 都来自注册表。
4. README 快捷键表可以通过 `npm run docs:shortcuts` 重新生成。
5. `npm run type-check` 通过。
6. 后续新增快捷键时，不需要手动修改多个 UI 文案位置。

## 8. 结论

当前 `04` 的主体目标已经完成，但 README 自动同步和 `Ctrl+Tab` 注册表纳入仍是长期维护风险。

完成本 `04.1` 后，快捷键体系才算真正闭环：所有快捷键都在注册表声明，所有用户可见说明都从注册表派生，README 可自动生成，后续不容易再次出现“文档和实际行为冲突”。
