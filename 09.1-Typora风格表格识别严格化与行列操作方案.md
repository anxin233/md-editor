# 09.1-Typora风格表格识别严格化与行列操作方案

关联文档：

- `09-Typora风格右键菜单交互收口与功能补齐方案.md`
- `08.1-Typora风格右键菜单功能规划方案.md`

## 1. 背景

`09` 文档第 1～5 阶段已经完成了右键位置一致性、WYSIWYG 图片识别、粘贴替换、源码链接编辑、表格块识别与删除等能力。

验证后确认大方向正确，但第 5 阶段还存在一个需要在进入第 6 阶段前收口的问题：

当前源码模式的表格块识别仍偏宽。如果普通段落中包含 `|`，并且它紧贴 Markdown 表格，现有连续行扫描逻辑可能把该普通段落一并纳入表格块，导致“删除表格”误删相邻文本。

因此下一步不应立即实现表格行列操作，而应先完成“表格块识别严格化”。只有当表格范围定位足够可靠后，再进入第 6 阶段的行列增删和对齐操作。

## 2. 本方案目标

本方案分为两个连续阶段：

1. 第 5.1 阶段：严格化 Markdown 表格块识别，确保“删除表格”不会误删普通段落。
2. 第 6 阶段：在可靠表格范围基础上，实现 Typora 风格表格行列操作。

核心目标是：

```text
先保证识别和删除绝对可靠，再增加行列编辑能力。
```

## 3. 当前问题

### 3.1 表格候选行判断过宽

当前源码模式中，表格候选行大致以“是否包含 `|`”作为判断基础。

这种规则会误判以下内容：

```md
这是一段普通文本，包含 A | B 这样的内容。

| Name | Age |
| --- | --- |
| Alice | 18 |
```

如果普通文本和表格之间没有空行，或者连续扫描逻辑过宽，该普通文本可能被当成表格块的一部分。

### 3.2 表格块边界缺少结构约束

标准 GFM 表格至少应满足：

1. 有表头行。
2. 表头下一行是分隔行。
3. 分隔行每一列由 `---`、`:---`、`---:`、`:---:` 等组成。
4. 数据行列数应与表头基本一致，或至少不能明显偏离。

如果不校验这些结构约束，就很难安全地做删除行、插入列、设置对齐等操作。

### 3.3 第 6 阶段不能建立在弱识别之上

表格行列操作需要准确知道：

1. 当前点击的是哪一个表格。
2. 当前点击的是哪一行。
3. 当前点击的是哪一列。
4. 表头行、分隔行和数据行分别在哪。
5. 每一行拆分后的单元格数量。

如果表格块识别不严格，行列操作会进一步放大误删和破坏内容的风险。

## 4. 第 5.1 阶段：表格块识别严格化

### 4.1 目标

让源码模式只在真正命中合法 GFM 表格时，才返回完整表格块范围。

同时保留弱匹配行为：

1. 如果当前行看起来像表格，但无法确认合法表格块，可以显示“打开表格编辑器”。
2. 但不能显示或执行“删除表格”。
3. 不能把普通包含 `|` 的段落纳入删除范围。

### 4.2 涉及文件

建议主要修改：

- `src/utils/editorContextMenu.ts`

可能涉及：

- `src/components/editor/MarkdownSource.vue`
- 后续如果抽出纯函数测试，可新增 `src/utils/markdownTable.ts`

### 4.3 建议新增表格解析模型

建议不要继续只返回 `{ from, to }`，而是先建立结构化解析结果。

建议类型：

```ts
interface MarkdownTableCellPosition {
  rowIndex: number
  columnIndex: number
}

interface MarkdownTableRow {
  lineNumber: number
  from: number
  to: number
  raw: string
  cells: string[]
  kind: 'header' | 'separator' | 'body'
}

interface MarkdownTableBlock {
  from: number
  to: number
  startLine: number
  endLine: number
  columnCount: number
  rows: MarkdownTableRow[]
  currentCell?: MarkdownTableCellPosition
}
```

第 5.1 阶段至少需要 `from`、`to`、`startLine`、`endLine`、`columnCount`、`rows`。

`currentCell` 可以先预留给第 6 阶段使用。

### 4.4 严格识别规则

#### 4.4.1 分隔行必须存在且位置正确

合法表格必须存在分隔行，并且分隔行上方必须是表头行。

例如：

```md
| A | B |
| --- | --- |
| 1 | 2 |
```

其中：

1. `| A | B |` 是表头行。
2. `| --- | --- |` 是分隔行。
3. 分隔行必须紧跟表头行。

如果当前点击的是数据行，应向上找到最近的分隔行，再确认它上一行是合法表头行。

#### 4.4.2 分隔行列数必须与表头一致

合法：

```md
| A | B |
| --- | --- |
```

不合法：

```md
| A | B |
| --- |
```

如果列数不一致，不应返回可删除表格块。

#### 4.4.3 数据行列数应与表头基本一致

建议第一版采用严格规则：

1. 数据行列数必须等于表头列数。
2. 空单元格允许存在。
3. 行首和行尾是否有 `|` 都要兼容。

例如以下两种都可以视为 2 列表格：

```md
| A | B |
| --- | --- |
| 1 | 2 |
```

```md
A | B
--- | ---
1 | 2
```

但为了降低误判，第一版建议优先支持带边界 `|` 的标准写法。无边界写法可以后续增强。

#### 4.4.4 普通段落不能被并入表格块

以下内容中，第一行不能被纳入表格：

```md
这是一段普通说明：A | B
| A | B |
| --- | --- |
| 1 | 2 |
```

严格规则应从合法分隔行出发，而不是从任意包含 `|` 的行出发无脑扩展。

### 4.5 推荐算法

#### 4.5.1 从点击行定位候选表格

输入：

1. CodeMirror `Text` 文档。
2. 点击位置 `pos`。

步骤：

1. 获取点击行 `currentLine`。
2. 如果当前行不包含 `|`，直接返回 `null`。
3. 在当前行附近向上、向下寻找合法分隔行。
4. 对每个候选分隔行，校验：
   - 上一行存在。
   - 上一行是合法表头行。
   - 表头列数与分隔行列数一致。
5. 如果当前点击行位于该候选表格范围内，则返回表格块。

#### 4.5.2 从分隔行扩展表格块

找到合法分隔行后：

1. 表格起始行为表头行。
2. 表格第二行为分隔行。
3. 从分隔行下一行开始向下扫描数据行。
4. 只有列数匹配的行才纳入表格。
5. 遇到空行、普通段落、列数不匹配行时停止。

#### 4.5.3 删除范围

删除完整表格块时：

1. `from` 使用表头行起始位置。
2. `to` 使用最后一行结束位置。
3. 如果最后一行后面有一个换行符，可以一并删除。
4. 不删除表格上方或下方的普通文本。

### 4.6 弱匹配策略

为了不破坏当前体验，可以保留弱匹配：

1. 当前行包含 `|` 但不能确认合法 GFM 表格。
2. 菜单可显示“打开表格编辑器”。
3. 不显示“删除表格”。
4. 不进入第 6 阶段行列操作。

这样可以兼顾旧行为和安全性。

### 4.7 验收标准

必须通过以下用例：

#### 用例 1：标准表格可识别并删除

```md
| A | B |
| --- | --- |
| 1 | 2 |
```

预期：

1. 右键任意表格行显示“删除表格”。
2. 删除后只移除这三行。

#### 用例 2：普通包含 `|` 的段落不显示删除表格

```md
这不是表格：A | B
```

预期：

1. 不显示“删除表格”。
2. 可以保留“打开表格编辑器”或不显示表格菜单。

#### 用例 3：普通段落紧贴表格不被误删

```md
这是一段普通说明：A | B
| A | B |
| --- | --- |
| 1 | 2 |
```

预期：

1. 右键表格时只删除后三行。
2. 第一行普通说明保留。

#### 用例 4：列数不一致不执行删除表格

```md
| A | B |
| --- |
| 1 | 2 |
```

预期：

1. 不显示“删除表格”。
2. 不进入表格行列操作。

#### 用例 5：表格下方普通文本不被误删

```md
| A | B |
| --- | --- |
| 1 | 2 |
普通文本 | 不是表格
```

预期：

1. 删除表格后，`普通文本 | 不是表格` 保留。

## 5. 第 6 阶段：表格行列操作

### 5.1 进入条件

只有第 5.1 阶段完成后，才能进入第 6 阶段。

进入条件：

1. 源码模式合法表格块识别稳定。
2. “删除表格”不会误删普通段落。
3. 表格解析结果能返回当前表格行列结构。
4. WYSIWYG 模式表格删除仍基于可靠 `pmRange`。

### 5.2 目标

实现 Typora 风格表格右键菜单的核心行列操作。

建议第一批实现：

1. 插入行到上方。
2. 插入行到下方。
3. 插入列到左侧。
4. 插入列到右侧。
5. 删除当前行。
6. 删除当前列。
7. 删除表格。

第二批再实现：

1. 设置左对齐。
2. 设置居中对齐。
3. 设置右对齐。

### 5.3 源码模式实现方案

#### 5.3.1 需要知道当前单元格

在 CodeMirror 中，右键点击位置应转换为：

1. 当前表格块。
2. 当前 Markdown 行。
3. 当前列。

建议函数：

```ts
function parseMarkdownTableAt(doc: Text, pos: number): MarkdownTableBlock | null
```

返回值应包含：

1. 表格范围。
2. 所有行。
3. 当前点击的单元格位置。
4. 列数。

#### 5.3.2 插入行

插入行到上方：

1. 如果当前行是表头或分隔行，不建议插入到它们之间。
2. 如果右键在表头，可以插入到第一条数据行上方。
3. 新行单元格数量应等于 `columnCount`。
4. 新行内容默认为空。

示例：

```md
| A | B |
| --- | --- |
| 1 | 2 |
```

插入下方后：

```md
| A | B |
| --- | --- |
| 1 | 2 |
|  |  |
```

#### 5.3.3 插入列

插入列到左侧或右侧：

1. 表头行插入空表头单元格。
2. 分隔行插入默认 `---`。
3. 数据行插入空单元格。
4. 所有行必须同步插入，保持列数一致。

示例：

```md
| A | B |
| --- | --- |
| 1 | 2 |
```

在 B 左侧插入列后：

```md
| A |  | B |
| --- | --- | --- |
| 1 |  | 2 |
```

#### 5.3.4 删除行

删除当前行规则：

1. 不允许删除表头行。
2. 不允许删除分隔行。
3. 允许删除数据行。
4. 如果删除后没有数据行，保留表头和分隔行。

#### 5.3.5 删除列

删除当前列规则：

1. 至少保留一列。
2. 删除列时表头、分隔行、所有数据行同步删除。
3. 如果当前列无法定位，不显示删除列菜单。

#### 5.3.6 对齐方式

对齐方式通过分隔行实现：

```md
| 左对齐 | 居中 | 右对齐 |
| :--- | :---: | ---: |
```

建议第二批实现：

1. 左对齐：`:---`
2. 居中：`:---:`
3. 右对齐：`---:`
4. 默认：`---`

### 5.4 WYSIWYG 模式实现方案

WYSIWYG 表格操作不应通过 DOM 直接修改。

优先方案：

1. 使用 ProseMirror 表格能力或 Milkdown 提供的命令。
2. 如果当前依赖未暴露行列操作命令，可以先复用 TableEditor 作为过渡。
3. 删除表格继续使用 `pmRange`。

WYSIWYG 第 6 阶段建议分两步：

1. 先补菜单项，但对未实现的行列操作显示提示：“WYSIWYG 表格行列操作将在后续版本支持，请先使用表格编辑器”。
2. 后续引入真正 ProseMirror 表格命令后，再打开这些操作。

这样可以避免源码模式已经支持、WYSIWYG 模式却用脆弱方式修改 DOM。

### 5.5 菜单设计

表格右键菜单建议：

```text
打开表格编辑器
--------
插入行到上方
插入行到下方
插入列到左侧
插入列到右侧
--------
删除当前行
删除当前列
删除表格
--------
对齐方式 >
  左对齐
  居中对齐
  右对齐
```

第一批可以先隐藏“对齐方式”。

### 5.6 验收标准

#### 行操作

1. 右键数据行，插入上方行成功。
2. 右键数据行，插入下方行成功。
3. 删除数据行后表头和分隔行保留。
4. 不允许删除表头行和分隔行。

#### 列操作

1. 插入左侧列后所有行列数一致。
2. 插入右侧列后所有行列数一致。
3. 删除列后所有行列数一致。
4. 不允许删除最后一列。

#### 内容安全

1. 表格上方普通文本不受影响。
2. 表格下方普通文本不受影响。
3. 包含 `|` 的普通段落不会被表格操作修改。
4. 操作后仍是合法 Markdown 表格。

## 6. 推荐实施顺序

### 6.1 第一批：识别严格化

1. 抽出或重写 `findMarkdownTableBlockRange()`。
2. 增加 `parseMarkdownTableAt()`。
3. 只允许合法 GFM 表格返回删除范围。
4. 弱匹配行只显示“打开表格编辑器”，不显示“删除表格”。
5. 手动验证所有第 5.1 用例。

### 6.2 第二批：源码模式行列操作

1. 基于 `MarkdownTableBlock` 实现插入行。
2. 实现删除行。
3. 实现插入列。
4. 实现删除列。
5. 接入源码模式右键菜单。

### 6.3 第三批：WYSIWYG 表格菜单对齐

1. WYSIWYG 保留“打开表格编辑器”和“删除表格”。
2. 对行列操作先提示或禁用。
3. 后续评估 ProseMirror 表格命令后再补齐。

### 6.4 第四批：对齐方式

1. 修改分隔行。
2. 支持左对齐、居中、右对齐。
3. 接入二级菜单。

## 7. 测试建议

### 7.1 优先补纯函数测试

建议优先为以下逻辑补测试：

1. 表格行拆分。
2. 分隔行识别。
3. 表格块解析。
4. 当前单元格定位。
5. 插入行。
6. 删除行。
7. 插入列。
8. 删除列。
9. 设置列对齐。

这些逻辑不依赖 Vue 或 Electron，适合单独测试。

### 7.2 手动验收清单

至少覆盖：

1. 标准 2 列表格。
2. 标准 3 列表格。
3. 空单元格。
4. 表格上方普通文本包含 `|`。
5. 表格下方普通文本包含 `|`。
6. 非法表格。
7. 只有表头和分隔行、没有数据行的表格。
8. 最后一列删除限制。
9. 表头行删除限制。
10. 分隔行删除限制。

## 8. 不建议本轮实现的内容

本轮不建议做：

1. 合并单元格。
2. 表格样式设置。
3. 表格拖拽调整列宽。
4. 复杂转义场景中的 `|` 解析。
5. HTML 表格解析。
6. WYSIWYG 中直接 DOM 操作表格。

这些能力复杂度高，不适合作为当前表格右键菜单的下一步。

## 9. 完成标准

本方案完成后，应达到：

1. 源码模式只对合法 GFM 表格显示“删除表格”和行列操作。
2. 普通包含 `|` 的段落不会被误判为表格块。
3. 删除表格不会误删表格上方或下方文本。
4. 源码模式表格支持基础行列增删。
5. WYSIWYG 模式至少保留稳定的打开表格编辑器和删除表格能力。
6. 对尚未支持的 WYSIWYG 行列操作，不使用脆弱 DOM 修改兜底。

最终验收标准：

```text
表格右键操作只能作用于被严格识别的合法表格，不得误删或改写普通文本。
```
