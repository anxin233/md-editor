# 09-Typora风格右键菜单交互收口与功能补齐方案

关联文档：

- `08-右键菜单功能规划与分阶段落地方案.md`
- `08.1-Typora风格右键菜单功能规划方案.md`

## 1. 背景

当前项目已经完成了一版 Typora 风格右键菜单实现，包含：

1. 全局右键菜单状态管理。
2. 递归子菜单 UI。
3. 源码模式和 WYSIWYG 模式的右键菜单接线。
4. 基础编辑菜单、段落菜单、列表菜单、插入菜单。
5. 链接、图片、代码块、表格、任务列表等语义菜单的初步识别。
6. WYSIWYG 下新增部分右键专用命令，例如编辑链接、取消链接、删除代码块、删除表格、删除图片。

但在验证 `08.1` 实现时发现，当前实现距离“像 Typora 一样右键当前对象并就地操作”还有几个关键差距。

核心问题不是菜单项数量不足，而是右键菜单的识别位置、显示内容和最终执行对象还没有完全统一。

如果继续堆叠菜单功能，后续会出现“菜单显示正确，但操作对象错误”的体验问题。因此下一步应优先收口右键交互一致性，再逐步补齐图片、链接、表格、任务列表等专项能力。

## 2. 当前主要问题

### 2.1 源码模式右键语义检测依赖旧光标

当前源码模式中，右键菜单构建逻辑主要依赖 CodeMirror 当前 selection 的光标位置。

问题表现：

1. 光标在 A 段。
2. 鼠标右键点击 B 段中的链接、图片、表格或代码块。
3. 菜单仍可能按 A 段上下文判断。

这不符合 Typora 风格的“点哪里，菜单就针对哪里”。

### 2.2 WYSIWYG 菜单识别位置与执行位置不一致

WYSIWYG 模式中，菜单构建时使用了 `posAtCoords()` 来识别右键点击位置，但后续执行命令时，`MilkdownEditor.vue` 中的删除链接、删除表格、删除代码块、删除图片等逻辑仍依赖 ProseMirror 当前 selection。

问题表现：

1. 用户右键点击表格。
2. 菜单显示“删除表格”。
3. 点击后可能删除旧光标附近的表格，或者没有任何效果。

这类问题会直接破坏右键菜单的可信度。

### 2.3 WYSIWYG 图片识别和删除不可靠

当前图片识别主要通过向上查找 ProseMirror 祖先节点实现。

但图片通常是 inline leaf node，不一定会作为当前 resolved position 的祖先节点出现。因此：

1. 右键图片时可能识别不到图片。
2. 图片专项菜单可能不显示。
3. “删除图片”可能找不到 image node，导致无效果。

### 2.4 WYSIWYG 粘贴仍依赖 `document.execCommand('paste')`

`document.execCommand('paste')` 在现代浏览器和 Electron sandbox 环境中经常被限制。

当前源码模式已经使用 `navigator.clipboard.readText()` 处理文本粘贴，但 WYSIWYG 仍保留 `execCommand('paste')`，存在运行时不可用风险。

### 2.5 源码模式链接编辑能力未补齐

当前源码模式已经可以识别部分链接并提供打开、复制、取消链接等能力，但“编辑链接”还未形成完整闭环。

Typora 风格右键菜单中，链接编辑是高频能力，应作为第二阶段专项菜单的一部分补齐。

### 2.6 表格能力仍停留在基础入口

当前表格右键主要提供“打开表格编辑器”和部分删除能力。

但 Typora 风格表格菜单通常包括：

1. 插入行。
2. 插入列。
3. 删除行。
4. 删除列。
5. 删除表格。
6. 设置对齐方式。

当前实现还没有达到该阶段。

## 3. 总体目标

本轮方案的目标是：

1. 保证右键菜单的识别位置、显示内容、执行对象完全一致。
2. 让源码模式和 WYSIWYG 模式都满足“点哪改哪”。
3. 修复 WYSIWYG 图片、链接、表格、代码块等专项操作的执行可靠性。
4. 用更稳定的剪贴板实现替换 WYSIWYG 中不可靠的粘贴逻辑。
5. 分阶段补齐 Typora 风格专项菜单能力。

本轮不建议继续做大范围 UI 美化，也不建议一次性补完所有 Typora 菜单项。优先级应放在交互正确性和行为一致性上。

## 4. 开发原则

### 4.1 点哪改哪

右键菜单必须以鼠标右键点击位置为准，而不是旧光标位置。

该原则适用于：

1. 源码模式。
2. WYSIWYG 模式。
3. 链接。
4. 图片。
5. 表格。
6. 代码块。
7. 任务列表。

### 4.2 菜单识别与命令执行共用同一位置上下文

不能出现菜单构建用右键位置、命令执行用旧 selection 的分裂。

建议建立统一的右键上下文模型，例如：

```ts
interface EditorContextMenuTarget {
  mode: 'source' | 'wysiwyg'
  pos: number
  kind: string
  range?: { from: number; to: number }
  href?: string
  src?: string
}
```

菜单构建和菜单 action 都基于这个上下文执行。

### 4.3 优先修正行为，再补齐菜单项

如果右键位置不可靠，新增更多菜单项只会放大问题。

推荐顺序：

1. 先修右键位置。
2. 再修 WYSIWYG 图片。
3. 再修 WYSIWYG 粘贴。
4. 再补链接、表格、任务列表能力。

### 4.4 源码模式和 WYSIWYG 模式允许实现不同，但体验应一致

源码模式可以通过 Markdown 文本 range 修改。

WYSIWYG 模式可以通过 ProseMirror transaction 修改。

底层实现可以不同，但用户看到的能力和结果应尽量一致。

## 5. 第一阶段：右键位置一致性收口

### 5.1 目标

修复当前最核心的问题：右键菜单必须根据鼠标点击位置构建，并且后续命令也作用在同一位置。

### 5.2 源码模式改造

涉及文件：

- `src/components/editor/MarkdownSource.vue`
- `src/utils/editorContextMenu.ts`

建议改造：

1. `MarkdownSource.vue` 的 `contextmenu` 事件中，根据鼠标坐标计算 CodeMirror 文档位置。
2. 将该位置传给 `buildTyporaCodeMirrorMenu()`。
3. `buildTyporaCodeMirrorMenu()` 不再默认读取 `view.state.selection.main.head` 作为语义检测位置。
4. 对于无选区的场景，可以在右键时同步 selection 到点击位置。
5. 对于已有选区的场景，如果右键点在选区内，应保留选区；如果右键点在选区外，应移动光标到右键位置。

建议行为：

```text
有选区，右键点在选区内：
  保留选区，菜单针对选区和右键位置共同判断。

有选区，右键点在选区外：
  清空旧选区，光标移动到右键位置。

无选区：
  光标移动到右键位置。
```

### 5.3 WYSIWYG 模式改造

涉及文件：

- `src/components/editor/WysiwygEditor.vue`
- `src/components/editor/MilkdownEditor.vue`
- `src/utils/editorContextMenu.ts`

建议改造：

1. 右键时使用 `pmView.posAtCoords()` 获取点击位置。
2. 在打开菜单前，将 ProseMirror selection 同步到该位置。
3. 如果用户右键点在已有选区内，可以保留选区。
4. 如果用户右键点在已有选区外，应移动 selection 到右键位置。
5. WYSIWYG 专项命令优先使用右键上下文 target；如果没有 target，再回退到当前 selection。

### 5.4 验收标准

1. 光标在 A 段，右键 B 段链接，菜单识别 B 段链接。
2. 光标在 A 段，右键 B 段代码块，显示代码块专项菜单。
3. 右键 WYSIWYG 表格后点击“删除表格”，删除当前右键表格。
4. 右键 WYSIWYG 链接后点击“编辑链接”，编辑当前右键链接。
5. 有选中文本时，右键选区内部不丢失选区。
6. 有选中文本时，右键选区外部应切换到新的右键位置。

## 6. 第二阶段：WYSIWYG 图片识别与删除修正

### 6.1 目标

让 WYSIWYG 模式能够可靠识别图片，并对当前右键图片执行删除或复制路径等操作。

### 6.2 建议实现

涉及文件：

- `src/utils/editorContextMenu.ts`
- `src/components/editor/MilkdownEditor.vue`

建议识别策略：

1. 优先从 DOM target 判断是否点击到图片元素。
2. 使用 `posAtCoords()` 获取位置后，检查 `nodeAfter`。
3. 必要时检查 `nodeBefore`。
4. 检查当前位置附近 inline node 是否为 `image`。
5. 返回图片 node 的准确 range。

建议上下文信息：

```ts
interface ImageContextTarget {
  kind: 'image'
  from: number
  to: number
  src?: string
  alt?: string
}
```

删除图片时，不应再通过祖先节点查找 image，而应直接删除 target range。

### 6.3 菜单范围

本阶段建议先实现：

1. 复制图片路径。
2. 删除图片引用。

暂不实现：

1. 删除本地图片文件。
2. 图片缩放。
3. 图片替换。
4. 在系统资源管理器中显示。

### 6.4 验收标准

1. WYSIWYG 模式右键图片时显示图片专项菜单。
2. 点击“复制图片路径”能复制当前图片地址。
3. 点击“删除图片”只删除当前图片引用。
4. 不删除本地图片文件。
5. 不误删图片所在段落的其他文字。

## 7. 第三阶段：WYSIWYG 粘贴能力替换

### 7.1 目标

替换不可靠的 `document.execCommand('paste')`，让 WYSIWYG 右键粘贴在 Electron 环境下更稳定。

### 7.2 建议实现

涉及文件：

- `src/utils/contextMenuRegistry.ts`
- `src/components/editor/WysiwygEditor.vue`
- `src/components/editor/MilkdownEditor.vue`

建议做法：

1. 使用 `navigator.clipboard.readText()` 读取剪贴板文本。
2. 读取成功后，通过 ProseMirror transaction 插入文本。
3. 插入位置应为右键同步后的 selection。
4. 读取失败时调用 `editorStore.showStatusToast()` 提示。
5. 图片粘贴继续走现有 paste 事件中的图片保存流程，不和文本粘贴混在一起。

### 7.3 验收标准

1. WYSIWYG 右键“粘贴”可以插入剪贴板文本。
2. 粘贴位置与右键位置一致。
3. 剪贴板权限不足时给出提示。
4. 不再依赖 `document.execCommand('paste')`。

## 8. 第四阶段：源码模式链接专项菜单补齐

### 8.1 目标

补齐源码模式下链接编辑能力，使其接近 Typora 的链接右键体验。

### 8.2 菜单项

源码模式右键 Markdown 链接时建议提供：

1. 打开链接。
2. 编辑链接。
3. 复制链接地址。
4. 取消链接。

### 8.3 建议实现

涉及文件：

- `src/utils/editorContextMenu.ts`

建议做法：

1. 当前链接识别保留 `range`、`linkText`、`url`。
2. “编辑链接”弹出输入框，默认值为当前 URL。
3. 用户取消输入时不修改文档。
4. 输入为空时提示“链接地址不能为空”。
5. 输入合法后，只替换 Markdown 链接语法中的 URL。
6. “取消链接”替换为纯文本 `linkText`。
7. 图片语法 `![alt](url)` 不能被误识别为普通链接。

### 8.4 安全要求

打开外部链接仍应保持当前安全策略：

1. 只允许 `https://`。
2. 不允许 renderer 任意打开本地文件路径。
3. 不允许任意协议直通 `openExternal`。

### 8.5 验收标准

1. 源码模式右键 `[text](https://example.com)` 显示链接专项菜单。
2. 点击“编辑链接”只修改 URL，不破坏链接文本。
3. 点击“取消链接”后保留 `text`。
4. 图片链接不被误判为普通链接。

## 9. 第五阶段：表格删除与定位收口

### 9.1 目标

先让表格基础操作稳定，再进入行列级操作。

### 9.2 建议实现

涉及文件：

- `src/utils/editorContextMenu.ts`
- `src/components/editor/MilkdownEditor.vue`

源码模式建议：

1. 表格识别不能只判断当前行是否包含 `|`。
2. 应识别连续 Markdown 表格块。
3. 应校验表头分隔行，例如 `| --- | --- |`。
4. 删除表格时删除完整表格块。
5. 打开表格编辑器时尽量定位当前表格。

WYSIWYG 模式建议：

1. 使用右键 target 定位当前 table node。
2. 删除表格时直接删除 table node range。
3. 打开表格编辑器时基于当前 table 上下文。

### 9.3 验收标准

1. 源码模式右键表格任意单元格，显示表格菜单。
2. 删除表格时删除完整表格，而不是只删当前行。
3. WYSIWYG 模式右键表格后能删除当前表格。
4. 不误判普通包含 `|` 的段落为表格。

## 10. 第六阶段：表格行列操作

### 10.1 目标

在表格基础定位稳定后，补齐 Typora 风格的行列编辑能力。

### 10.2 菜单项

建议提供：

1. 插入行到上方。
2. 插入行到下方。
3. 插入列到左侧。
4. 插入列到右侧。
5. 删除当前行。
6. 删除当前列。
7. 设置列对齐。
8. 删除表格。

### 10.3 实现要求

源码模式：

1. 使用结构化 Markdown 表格解析逻辑。
2. 修改行列时保持表头、分隔行、单元格数量一致。
3. 尽量保留原表格文本风格。

WYSIWYG 模式：

1. 优先使用 ProseMirror 表格相关能力。
2. 如果当前 Milkdown 表格能力不足，应先通过 TableEditor 兜底。
3. 不建议用脆弱 DOM 操作直接修改表格。

### 10.4 验收标准

1. 插入行列后仍是合法 Markdown 表格。
2. 删除行列后不破坏表头分隔行。
3. 删除最后一列或最后一行时应有合理限制或提示。
4. 源码模式和 WYSIWYG 模式行为一致，或差异有明确提示。

## 11. 第七阶段：任务列表与列表增强

### 11.1 目标

补齐 Typora 风格中常用的任务列表和列表操作。

### 11.2 任务列表菜单

右键任务列表项时建议提供：

1. 标记为完成 / 标记为未完成。
2. 转为普通列表。
3. 转为有序列表。
4. 增加缩进。
5. 减少缩进。

### 11.3 普通列表菜单

右键普通列表项时建议提供：

1. 转为无序列表。
2. 转为有序列表。
3. 转为任务列表。
4. 退出列表。
5. 增加缩进。
6. 减少缩进。

### 11.4 验收标准

1. 右键 `- [ ] task` 可切换为 `- [x] task`。
2. 右键 `- [x] task` 可切换为 `- [ ] task`。
3. 普通列表可转换为任务列表。
4. 列表缩进和反缩进不破坏列表结构。
5. 源码模式和 WYSIWYG 模式结果一致。

## 12. 推荐执行顺序

建议严格按以下顺序推进：

1. 右键位置一致性修正。
2. WYSIWYG 图片识别与删除。
3. WYSIWYG 粘贴替换。
4. 源码模式链接编辑。
5. 表格删除与定位。
6. 表格行列操作。
7. 任务列表与列表缩进增强。

不建议先做表格行列操作，也不建议先扩展更多视觉菜单。原因是当前最关键的基础问题是右键位置上下文不统一。

## 13. 测试建议

### 13.1 手动测试用例

准备一个 Markdown 文档，包含：

```md
# 标题

普通段落，包含 [链接](https://example.com)。

![图片](./images/demo.png)

```js
console.log('hello')
```

| A | B |
| --- | --- |
| 1 | 2 |

- [ ] 未完成任务
- [x] 已完成任务

> 引用内容
```

测试场景：

1. 光标在标题，右键段落中的链接。
2. 光标在段落，右键代码块。
3. 光标在段落，右键表格。
4. 光标在段落，右键图片。
5. 选中文本后右键选区内部。
6. 选中文本后右键选区外部。
7. WYSIWYG 模式右键图片并删除。
8. WYSIWYG 模式右键粘贴文本。
9. 源码模式右键编辑链接。
10. 源码模式右键取消链接。

### 13.2 自动化验证建议

如果后续补测试，建议优先覆盖纯函数：

1. `detectCodeMirrorContext()`。
2. Markdown 链接识别。
3. Markdown 图片识别。
4. Markdown 表格块识别。
5. 表格行列修改函数。
6. 任务列表切换函数。

对于 ProseMirror 相关行为，可以先保留手动验收，后续再引入编辑器层测试。

## 14. 风险与边界

### 14.1 Markdown 语法边界

源码模式识别链接、图片、表格、代码块时，不应追求一次覆盖所有极端 Markdown 写法。

建议优先覆盖常见写法：

1. 标准行内链接。
2. 标准 Markdown 图片。
3. 标准围栏代码块。
4. 标准 GFM 表格。
5. 标准任务列表。

极端嵌套和非标准语法可以后续增强。

### 14.2 WYSIWYG 与源码模式差异

部分能力在 WYSIWYG 中更适合通过 ProseMirror transaction 实现，在源码模式中更适合通过文本 range 修改实现。

两边不需要强行共用同一个底层函数，但应共用同一产品语义和验收标准。

### 14.3 表格能力复杂度

表格行列操作是本方案中复杂度最高的部分。

不建议和右键位置修正一起开发。应先保证表格识别和删除稳定，再逐步引入插入行列、删除行列、对齐方式等能力。

## 15. 完成标准

本方案完成后，应达到以下效果：

1. 右键菜单严格跟随鼠标点击位置。
2. 菜单显示内容和最终执行对象一致。
