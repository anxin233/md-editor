# 05-文件访问授权边界与导出资源安全方案

关联问题：`01-项目审查报告.md` 中的 `3.4 文件授权边界偏松` 与 `3.5 导出本地图片可能读取任意本地文件`

## 1. 问题背景

当前项目已经使用了 Electron 的安全基础配置：

- `nodeIntegration: false`
- `contextIsolation: true`
- `sandbox: true`
- preload 只暴露封装后的 `window.electronAPI`
- 主进程文件读写有 `allowedPaths` 白名单

但现有白名单模型仍有一个关键漏洞：renderer 可以主动调用 `file.authorize(targetPath)`，把任意路径加入主进程允许列表。

相关代码：

- `electron/preload/index.ts` 暴露 `file.authorize`
- `electron/main/ipc/file.ipc.ts` 注册 `file:grantAccess`
- `src/stores/file.ts` 在打开、保存前主动 `authorizePath(filePath)`
- `src/utils/export.ts` 导出图片前主动 `file.authorize(absolutePath)` 再 `readBinary`
- `src/stores/settings.ts` 加载自定义 CSS 前主动 `file.authorize(cssPath)`

这意味着只要 renderer 被 XSS、恶意 Markdown 内容、未来插件、供应链脚本或其他不可信代码影响，就可能扩大本地文件读写范围。

## 2. 当前风险

### 2.1 renderer 可以授予自己任意路径权限

当前 `file:grantAccess`：

```ts
ipcMain.handle('file:grantAccess', (_event, targetPath: string): boolean => {
  const safeTarget = resolve(targetPath)
  addAllowedPath(safeTarget)
  addAllowedPath(dirname(safeTarget))
  return true
})
```

问题在于：

1. 调用方来自 renderer。
2. 主进程没有验证路径是否来自系统对话框、工作区、已打开文件或用户确认。
3. 还会同时授权目标路径和父目录。

最终结果是，白名单机制存在，但授权入口不可信。

### 2.2 导出流程会尝试读取任意本地图片路径

当前 `inlineLocalImages()` 会处理 Markdown 渲染后的 `<img src>`：

```ts
await window.electronAPI?.file.authorize(absolutePath)
const bytes = await window.electronAPI?.file.readBinary(absolutePath)
```

风险场景：

```md
![x](C:/Users/ASUS/.ssh/id_rsa)
```

或：

```html
<img src="file:///C:/Users/ASUS/Desktop/private.txt">
```

虽然这些内容未必是合法图片，但当前逻辑可能把文件内容读取为二进制，再作为 data URL 写入导出结果。

### 2.3 最近文件和自定义 CSS 也不能直接信任 renderer

当前最近文件由 renderer 调用 `recent:add` 写入，后续如果把“最近文件”当作可信授权来源，需要同步收紧 recent IPC。

自定义 CSS 当前路径保存在 renderer 的 `localStorage` 中，重启后通过 `loadCustomCss(savedCustomCss)` 加载。如果继续保留 renderer 任意授权，风险仍然存在；如果移除任意授权，则需要重新设计自定义 CSS 的持久授权方式。

## 3. 方案目标

本方案目标不是临时拦截某个路径，而是重建文件访问模型：

1. renderer 不再拥有“给自己授权任意路径”的能力。
2. 授权只来自可信入口：系统对话框、已打开工作区、当前文档目录、主进程持久化的可信记录。
3. 主进程统一判断“某个路径能否被某个操作访问”。
4. 导出本地资源时只读取安全范围内的图片资源。
5. 图片资源必须满足扩展名、MIME、文件大小和路径范围限制。
6. 最近文件、自定义 CSS 等功能不再依赖 renderer 侧任意授权。

## 4. 总体设计

### 4.1 移除公开授权 API

从 preload 暴露层移除：

```ts
file.authorize(targetPath)
```

从主进程移除公开 IPC：

```ts
file:grantAccess
```

保留主进程内部函数，例如：

```ts
grantFilePermission(...)
grantDirectoryPermission(...)
```

这些函数只能由主进程可信流程调用，不能暴露给 renderer。

### 4.2 从“路径白名单”升级为“权限条目”

当前 `allowedPaths` 只是 `Set<string>`，只能表达“某路径及其子路径可访问”。

建议改为结构化权限：

```ts
type FileOperation =
  | 'readText'
  | 'readBinary'
  | 'writeText'
  | 'readDir'
  | 'create'
  | 'rename'
  | 'delete'
  | 'watch'
  | 'saveImage'

interface PermissionEntry {
  root: string
  kind: 'file' | 'directory'
  source: 'open-file-dialog' | 'open-folder-dialog' | 'save-file-dialog' | 'css-dialog' | 'recent-file'
  operations: Set<FileOperation>
}
```

然后所有文件 IPC 都通过：

```ts
validatePath(filePath, operation)
```

而不是只判断路径是否在 `allowedPaths` 中。

### 4.3 授权来源规则

#### 打开文件

`dialog:openFile` 成功后：

- 授权该文件：
  - `readText`
  - `writeText`
  - `watch`
- 授权该文件所在目录下的文档资源读取：
  - 仅用于导出和预览相关的图片读取
  - 不应等同于完整读写目录权限

#### 打开文件夹

`dialog:openFolder` 成功后：

- 授权该目录递归：
  - `readDir`
  - `readText`
  - `writeText`
  - `create`
  - `rename`
  - `delete`
  - `watch`
  - `saveImage`

这是工作区模式的可信入口。

#### 保存文件 / 另存为

`dialog:saveFile` 成功后：

- 授权目标文件：
  - `writeText`
  - `readText`
  - `watch`
- 如果是新建文档，允许其父目录用于保存粘贴图片：
  - `saveImage`

#### 自定义 CSS

`dialog:openCssFile` 成功后：

- 只授权该 CSS 文件：
  - `readText`

不要授权整个 CSS 文件所在目录的任意读写。

#### 最近文件

最近文件不能继续由 renderer 任意写入后直接信任。

建议调整为：

1. `recent:add` 不再作为公开 renderer 可随意传路径的接口。
2. 最近文件只由主进程在“打开文件成功”或“保存成功”后写入。
3. 如果保留 `recent:add`，主进程必须校验该路径当前已经拥有合法权限，否则拒绝写入。
4. 打开最近文件时，主进程先确认该路径来自主进程维护的最近文件列表，再授予该文件最小权限。

## 5. 文件 IPC 改造方案

### 5.1 主进程权限模块

建议把文件权限逻辑从 `file.ipc.ts` 中拆出：

```text
electron/main/security/fileAccess.ts
```

职责：

- 规范化路径。
- 管理权限条目。
- 校验路径是否属于某个授权文件或目录。
- 校验操作是否被允许。
- 提供文档资源路径解析和图片安全校验。

推荐导出：

```ts
grantFilePermission(filePath, options)
grantDirectoryPermission(dirPath, options)
validateFileAccess(targetPath, operation)
isPathInside(parentDir, targetPath)
```

### 5.2 file.ipc.ts 只保留业务文件操作

`file.ipc.ts` 中的每个 handler 都必须声明操作类型：

```ts
ipcMain.handle('file:read', async (_event, filePath: string) => {
  const safePath = validateFileAccess(filePath, 'readText')
  return readFile(safePath, 'utf-8')
})

ipcMain.handle('file:write', async (_event, filePath: string, content: string) => {
  const safePath = validateFileAccess(filePath, 'writeText')
  await writeFile(safePath, content, 'utf-8')
  return true
})
```

`file:readBinary` 不建议继续作为通用公开能力暴露给 renderer。

如果仍需保留，也只能用于明确的安全资源读取场景，并限制操作来源。

### 5.3 saveImage 限制

当前 `saveImage(dirPath, fileName, buffer)` 已经会把图片放到 `images` 目录，但还需要补充：

1. `dirPath` 必须有 `saveImage` 权限。
2. `fileName` 必须是纯文件名，不能包含路径分隔符。
3. 扩展名只允许：
   - `.png`
   - `.jpg`
   - `.jpeg`
   - `.gif`
   - `.webp`
4. 文件大小限制，例如 10 MB。
5. 文件头 MIME 检测，不能只相信扩展名。

推荐逻辑：

```ts
const safeDir = validateFileAccess(dirPath, 'saveImage')
const safeName = sanitizeImageFileName(fileName)
assertAllowedImageExtension(safeName)
assertImageSize(buffer)
assertImageSignature(buffer, safeName)
```

## 6. 导出本地图片安全方案

### 6.1 不再在 renderer 中 authorize + readBinary

当前导出逻辑的问题是 renderer 自己决定：

1. 哪个路径需要授权。
2. 哪个路径需要读取。

建议改成主进程提供专用接口：

```ts
file.readDocumentImage(sourceFilePath, imageSrc)
```

返回：

```ts
interface DocumentImageResult {
  bytes: Uint8Array
  mime: string
}
```

renderer 只负责把 Markdown 渲染出的 `img.src` 交给主进程，主进程决定是否允许读取。

### 6.2 只允许相对图片资源

导出时允许：

```md
![a](./images/a.png)
![a](images/a.png)
![a](assets/a.webp)
```

默认拒绝：

```md
![a](C:/Users/ASUS/private.txt)
![a](/etc/passwd)
![a](file:///C:/Users/ASUS/private.txt)
![a](../private.txt)
```

`http://`、`https://`、`data:`、`blob:` 不走本地读取：

- `data:` 保持原样。
- `http/https` 保持原样，是否联网加载由导出渲染环境决定。
- `blob:` 通常导出时不可复现，保持原样或跳过。

### 6.3 路径解析必须在主进程做

主进程根据 `sourceFilePath` 得到文档目录：

```ts
const documentDir = dirname(sourceFilePath)
```

再解析相对图片：

```ts
const resolved = resolve(documentDir, imageSrc)
```

然后验证：

```ts
isPathInside(documentDir, resolved)
```

路径穿越必须拒绝：

```md
![a](../../secret.png)
```

### 6.4 图片类型限制

主进程读取前先看扩展名：

```ts
const allowed = new Set(['.png', '.jpg', '.jpeg', '.gif', '.webp', '.svg'])
```

建议第一版谨慎处理 SVG：

- 如果保留 SVG，需要清理脚本和事件属性。
- 如果不做 SVG 清理，导出内联阶段可以先禁用 SVG 本地内联。

推荐第一版只允许：

- PNG
- JPEG
- GIF
- WEBP

SVG 暂时不内联，保留原路径。

### 6.5 文件大小限制

建议限制单张图片最大 10 MB，总内联图片最大 50 MB。

原因：

- 防止误读大文件导致卡顿。
- 防止导出文件体积暴涨。
- 防止恶意 Markdown 构造大量本地资源拖垮导出。

### 6.6 导出失败行为

如果某张图片不符合规则：

- 不抛出中断整个导出的错误。
- 保持原 `src`。
- 记录 warning。

后续可在导出完成后提示：

```text
部分本地图片因路径不安全或文件过大，未被内联。
```

这也可以和后续 `4.7 导出体验可增强` 的全局导出状态一起处理。

## 7. Renderer 侧调用改造

### 7.1 file store

当前：

```ts
async function authorizePath(filePath: string) {
  await window.electronAPI?.file.authorize(filePath)
}
```

建议删除 `authorizePath()`。

打开文件流程改为：

1. 用户通过 `dialog.openFile()` 选择文件。
2. 主进程在 dialog handler 内完成授权。
3. renderer 调用 `file.read(filePath)`。

保存流程改为：

1. 如果是新文件或另存为，先 `dialog.saveFile()`。
2. 主进程在 dialog handler 内授权保存目标。
3. renderer 调用 `file.write(filePath, content)`。

打开最近文件流程不能直接调用 `file.read(path)` 后自动授权，应走新的主进程受控接口，例如：

```ts
recent.open(filePath)
```

或：

```ts
file.openRecent(filePath)
```

主进程确认该路径属于可信最近文件列表后，再授权并读取。

### 7.2 settings store

当前 `loadCustomCss(cssPath)` 会主动授权路径。

建议拆分两个入口：

```ts
pickAndLoadCustomCss()
loadPersistedCustomCss()
```

`pickAndLoadCustomCss()`：

1. 调用 `dialog.openCssFile()`。
2. 主进程授权该 CSS 文件。
3. 调用 `file.read(cssPath)`。
4. 主进程把该 CSS 路径写入主进程受控设置。

`loadPersistedCustomCss()`：

1. 从主进程设置读取已授权 CSS 路径。
2. 主进程确认该路径确实来自此前的 CSS dialog。
3. 再允许读取。

如果暂时不做主进程设置持久化，则重启后要求用户重新选择 CSS，比继续保留 renderer 任意授权更安全。

### 7.3 export.ts

当前：

```ts
await window.electronAPI?.file.authorize(absolutePath)
const bytes = await window.electronAPI?.file.readBinary(absolutePath)
```

改为：

```ts
const image = await window.electronAPI?.file.readDocumentImage(sourceFilePath, src)
if (!image) continue
img.setAttribute('src', `data:${image.mime};base64,${toBase64(image.bytes)}`)
```

`resolveLocalAssetPath()` 中对绝对路径、`file:` URL 的支持应删除或只用于识别后跳过，不应作为可读取路径。

## 8. Preload API 调整

### 8.1 删除

从 `ElectronAPI.file` 删除：

```ts
authorize(targetPath)
readBinary(filePath)
```

如果 `readBinary` 后续只有导出图片使用，应彻底替换为专用接口。

### 8.2 新增

建议新增：

```ts
readDocumentImage: (
  sourceFilePath: string,
  imageSrc: string
) => Promise<{ bytes: Uint8Array; mime: string } | null>
```

可选新增：

```ts
openRecentFile: (filePath: string) => Promise<{ filePath: string; fileName: string; content: string } | null>
```

或者放到 `recent` namespace：

```ts
recent.open(filePath)
```

## 9. 主进程最近文件方案

当前 `recent:add(filePath, fileName)` 可由 renderer 任意传入。

建议改为：

1. `recent:get` 保留。
2. `recent:remove`、`recent:clear` 保留。
3. `recent:add` 不再公开，或仅允许添加当前已授权路径。
4. 打开文件、保存文件成功后，由主进程内部调用 `addRecentFile()`。
5. 新增 `recent:open`：
   - 检查路径是否存在于主进程 recent 列表。
   - 检查文件仍存在。
   - 授权该文件最小权限。
   - 返回文件内容。

这样 renderer 不能通过伪造 recent 记录绕过授权。

## 10. 推荐实现顺序

### 第一阶段：建立权限模型

1. 新增 `electron/main/security/fileAccess.ts`。
2. 用 `PermissionEntry` 替换简单 `allowedPaths`。
3. 实现 `grantFilePermission()`、`grantDirectoryPermission()`、`validateFileAccess()`。
4. 保持现有功能暂时不变，只把主进程内部校验替换为新模型。

### 第二阶段：移除 renderer 任意授权

1. 删除 preload 中的 `file.authorize`。
2. 删除 `file:grantAccess` IPC。
3. 删除 `src/stores/file.ts` 中的 `authorizePath()` 调用。
4. 删除 `src/stores/settings.ts` 中的 `file.authorize(cssPath)`。
5. 删除 `src/utils/export.ts` 中的 `file.authorize(absolutePath)`。

### 第三阶段：补齐可信入口授权

1. `dialog:openFile` 授权打开的文件。
2. `dialog:openFolder` 授权工作区目录。
3. `dialog:saveFile` 授权保存目标。
4. `dialog:openCssFile` 只授权 CSS 文件。
5. 图片粘贴保存路径只允许当前文档目录或工作区目录。

### 第四阶段：导出图片安全收口

1. 删除通用 `file.readBinary` 暴露。
2. 新增 `file.readDocumentImage(sourceFilePath, imageSrc)`。
3. 主进程限制图片必须是当前文档目录下的相对资源。
4. 限制扩展名、文件头和大小。
5. 导出时对不允许读取的图片保持原路径并记录 warning。

### 第五阶段：最近文件与自定义 CSS 收口

1. 最近文件由主进程维护，不再信任 renderer 任意 add。
2. 打开最近文件走主进程 `recent.open`。
3. 自定义 CSS 的持久路径改为主进程保存，或重启后要求用户重新选择。

### 第六阶段：测试与验证

1. 为 `fileAccess.ts` 增加单元测试。
2. 覆盖路径穿越、绝对路径、工作区路径、文件权限、目录权限。
3. 覆盖导出图片：
   - 相对图片可内联。
   - `../` 越界图片不可内联。
   - 绝对路径不可内联。
   - 非图片扩展名不可内联。
   - 超大图片不可内联。

## 11. 验收标准

完成后应满足：

1. `window.electronAPI.file.authorize` 不再存在。
2. `file:grantAccess` IPC 不再存在。
3. renderer 无法通过传入任意路径扩大访问权限。
4. 文件读写、目录读取、删除、重命名、watch 都经过带 operation 的权限校验。
5. 打开文件、打开文件夹、保存文件、自定义 CSS 选择都由主进程可信流程授予最小权限。
6. 导出本地图片不再读取绝对路径或 `file:` URL。
7. 导出只内联当前文档目录下的安全相对图片。
8. 图片内联有扩展名、MIME 和大小限制。
9. 最近文件不能被 renderer 伪造为授权来源。
10. `npm run type-check` 通过。

## 12. 需要特别注意的兼容点

### 12.1 打开最近文件

移除 renderer 任意授权后，打开最近文件不能再依赖：

```ts
authorizePath(filePath)
file.read(filePath)
```

必须新增主进程受控入口，否则最近文件会失效。

### 12.2 自定义 CSS 重启加载

当前 CSS 路径保存在 renderer 的 `localStorage`。如果移除任意授权，重启后自动读取 CSS 会被拒绝。

推荐改为主进程持久化“用户通过 CSS 对话框选择过的 CSS 文件”，不要继续信任 renderer localStorage。

### 12.3 图片粘贴

源码模式和 WYSIWYG 模式都会调用 `file.saveImage(basePath, fileName, buffer)`。

`basePath` 应来自当前已保存文档的目录。主进程要确认这个目录确实是当前文档目录或工作区目录，不能只相信 renderer 传入。

### 12.4 工作区操作

打开文件夹后，工作区内的新增、重命名、删除是合理能力，但必须限定在工作区目录内。

不要因为支持工作区操作，就重新把磁盘任意路径开放给 renderer。

## 13. 不建议的做法

不建议只做以下临时修补：

1. 在 `export.ts` 里简单判断一下扩展名，但继续保留 `file.authorize`。
2. 只禁止导出绝对路径，但保留 renderer 任意授权。
3. 继续让最近文件由 renderer 任意写入，然后把 recent 当成可信来源。
4. 只在 renderer 里做路径判断，不在主进程校验。

这些做法都没有解决根问题：主进程授权入口不可信。

## 14. 结论

建议把 `3.4` 和 `3.5` 合并处理，作为一次文件访问安全边界重构。

优先级高于普通体验问题，因为它决定了这个 Electron 应用未来能否安全承载 Markdown 预览、导出、自定义 CSS、插件等能力。

最终目标是：renderer 只能“请求操作”，不能“授予权限”；真正的路径授权、范围判断和本地资源读取都由主进程集中控制。
