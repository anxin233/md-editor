# 08-右键菜单功能规划与分阶段落地方案

关联背景：当前项目缺少右键菜单能力，用户在编辑、文件管理、标签管理、预览阅读等场景中需要频繁移动到顶部菜单或使用快捷键，桌面端体验不完整。

## 1. 产品目标

右键菜单不是简单补一个通用菜单，而是要按上下文提供“当前最可能需要的操作”。

核心目标：

1. 降低常用操作路径，让写作和文件管理更顺手。
2. 补齐桌面端编辑器的基础交互预期。
3. 让不同区域的菜单符合用户心智。
4. 与已有顶部菜单、快捷键、命令体系保持一致。
5. 避免右键菜单变成“所有功能堆叠”，保持轻量、高频、准确。

## 2. 设计原则

### 2.1 上下文感知

不同区域展示不同菜单：

- 编辑区：文本编辑与格式化。
- 文件树：文件 / 文件夹管理。
- 标签栏：标签和保存相关操作。
- 预览区：阅读、复制、打开链接。
- 图片：图片查看、复制、替换、删除引用。

### 2.2 和现有命令体系复用

右键菜单不应重新实现一套业务逻辑。

推荐复用：

- `appCommands.ts`
- `shortcutRegistry.ts`
- 文件 store
- 设置 store
- 已有格式化命令
- 已有文件 IPC 能力

这样可以避免出现“顶部菜单可用、右键不可用”或“快捷键行为和右键行为不一致”的问题。

### 2.3 不可用项禁用或隐藏

不同编辑模式能力不同。

例如：

- WYSIWYG 暂不支持的命令不要显示为可点击。
- 没有选中文本时，不显示“剪切”“删除选中内容”。
- 未保存文件不显示“复制文件路径”。
- 非图片节点不显示图片菜单。

### 2.4 危险操作必须确认

以下操作必须有确认或复用现有关闭保存流程：

- 删除文件
- 删除文件夹
- 关闭未保存标签
- 关闭其他未保存标签
- 删除图片引用

### 2.5 菜单保持轻量

单层菜单建议控制在 8 到 12 个主项以内。

低频能力可以放入二级菜单，或只放在顶部菜单中，不必全部塞进右键菜单。

## 3. 菜单场景规划

## 3.1 编辑区右键菜单

适用范围：

- 源码模式编辑区
- WYSIWYG 编辑区
- 分栏模式中的源码编辑区

### 有选中文本时

建议菜单项：

1. 剪切
2. 复制
3. 粘贴
4. 删除
5. 全选
6. 加粗
7. 斜体
8. 删除线
9. 行内代码
10. 超链接
11. 引用
12. 有序列表
13. 无序列表
14. 复制为 Markdown
15. 复制为纯文本

优先级说明：

- 第一组是系统级文本操作，必须优先。
- 第二组是 Markdown 高频格式化。
- “复制为 Markdown / 纯文本”是进阶能力，可以第二阶段做。

### 无选中文本时

建议菜单项：

1. 粘贴
2. 全选
3. 插入图片
4. 插入链接
5. 插入表格
6. 插入代码块
7. 插入公式块
8. 插入分割线
9. 插入 TOC 标记
10. 插入 YAML Front Matter

### 模式差异

源码模式：

- 可以直接插入 Markdown 语法。
- 可以使用字符串级包装和插入。

WYSIWYG 模式：

- 应优先使用 Milkdown / ProseMirror 命令。
- 暂不支持的命令应隐藏或禁用。
- 不应通过不可靠的字符串 `indexOf` 反查选区。

## 3.2 文件树右键菜单

适用范围：

- 左侧资源管理器文件节点
- 文件夹节点
- 文件树空白区域

### 文件右键

建议菜单项：

1. 打开
2. 在新标签打开
3. 重命名
4. 删除
5. 复制路径
6. 在系统资源管理器中显示
7. 导出为 PDF
8. 导出为 HTML
9. 导出为 Word

第一期建议先做：

- 打开
- 重命名
- 删除
- 复制路径
- 刷新父目录

### 文件夹右键

建议菜单项：

1. 新建文件
2. 新建文件夹
3. 重命名
4. 删除
5. 刷新
6. 复制路径
7. 在系统资源管理器中显示

### 空白区域右键

建议菜单项：

1. 新建文件
2. 新建文件夹
3. 刷新工作区
4. 打开文件夹

### 文件树注意事项

删除逻辑必须避免“先关闭标签，删除失败导致 UI 状态丢失”。

推荐规则：

1. 先确认用户意图。
2. 执行主进程删除。
3. 删除成功后再关闭相关标签和刷新文件树。
4. 删除失败时保持标签和文件树状态不变，并提示错误。

## 3.3 标签页右键菜单

适用范围：

- 顶部标签栏
- 单个标签页

建议菜单项：

1. 保存
2. 另存为
3. 关闭
4. 关闭其他标签
5. 关闭右侧标签
6. 关闭已保存标签
7. 复制文件路径
8. 在系统资源管理器中显示

未保存新文件时：

1. 保存
2. 另存为
3. 关闭
4. 关闭其他标签

标签页右键的价值很高，尤其是：

- 关闭其他标签
- 关闭右侧标签
- 关闭已保存标签

这些是多标签编辑器的典型高频能力。

## 3.4 预览区右键菜单

适用范围：

- 分栏模式预览区
- 后续如果支持独立预览，也适用

普通内容右键：

1. 复制
2. 全选
3. 复制为纯文本
4. 复制为 Markdown

链接右键：

1. 打开链接
2. 复制链接地址

代码块右键：

1. 复制代码
2. 复制代码块 Markdown

预览区菜单要偏“阅读与复制”，不建议混入大量编辑命令。

## 3.5 图片右键菜单

适用范围：

- 编辑区中的图片
- 预览区中的图片

建议菜单项：

1. 查看原图
2. 复制图片
3. 复制图片路径
4. 在系统资源管理器中显示
5. 替换图片
6. 设置替代文本
7. 删除图片引用

图片是 Markdown 编辑器的高频痛点，建议作为第二阶段重点能力。

注意：

- 删除图片引用只删除 Markdown 引用，不默认删除本地图片文件。
- 如果要删除本地图片文件，必须额外确认。
- 替换图片要保持相对路径策略，不要绕过现有文件授权模型。

## 4. 分阶段落地方案

### 第一阶段：基础右键菜单

目标：先补齐桌面应用最基础的右键体验。

范围：

1. 编辑区基础菜单：
   - 剪切
   - 复制
   - 粘贴
   - 删除
   - 全选
2. 文件树基础菜单：
   - 打开
   - 新建文件
   - 新建文件夹
   - 重命名
   - 删除
   - 刷新
   - 复制路径
3. 标签页基础菜单：
   - 保存
   - 关闭
   - 关闭其他标签
   - 复制文件路径

验收标准：

- 在编辑区右键能完成系统文本操作。
- 文件树中文件 / 文件夹 / 空白区域能展示不同菜单。
- 标签页右键能关闭当前和关闭其他。
- 删除文件失败时不丢失已打开标签状态。

### 第二阶段：Markdown 编辑增强

目标：让右键菜单服务写作和排版。

范围：

1. 加粗
2. 斜体
3. 删除线
4. 行内代码
5. 超链接
6. 引用
7. 有序列表
8. 无序列表
9. 插入表格
10. 插入代码块
11. 插入公式块
12. 插入 TOC 标记
13. 插入 YAML Front Matter

验收标准：

- 源码模式右键格式化与顶部菜单行为一致。
- WYSIWYG 模式只展示或启用已支持命令。
- 右键格式化命令复用现有命令体系，不单独维护另一套行为。

### 第三阶段：预览和图片菜单

目标：提升阅读、复制和图片管理体验。

范围：

1. 预览区复制文本。
2. 链接打开和复制链接。
3. 代码块复制。
4. 图片查看原图。
5. 图片复制路径。
6. 图片替换。
7. 删除图片引用。

验收标准：

- 右键链接时能识别链接上下文。
- 右键代码块时能复制代码内容。
- 右键图片时显示图片专属菜单。
- 图片操作不绕过文件访问安全边界。

### 第四阶段：高级能力

目标：让右键菜单可扩展、可维护。

范围：

1. 菜单项快捷键提示。
2. 根据用户设置隐藏低频菜单项。
3. 插件扩展菜单预留接口。
4. 最近操作智能推荐。

这一阶段不建议过早实施，等核心菜单稳定后再做。

## 5. 技术实现建议

### 5.1 建立统一 Context Menu Registry

建议新增：

```text
src/utils/contextMenuRegistry.ts
```

用于定义：

- 菜单 id
- 菜单标题
- 所属场景
- 是否启用
- 对应命令
- 是否危险操作
- 是否需要二次确认

不要在每个组件中硬编码菜单项。

### 5.2 菜单场景类型

建议定义：

```ts
type ContextMenuScope =
  | 'editor'
  | 'file-tree-file'
  | 'file-tree-folder'
  | 'file-tree-empty'
  | 'tab'
  | 'preview'
  | 'link'
  | 'image'
  | 'code-block'
```

### 5.3 菜单项结构

建议结构：

```ts
interface ContextMenuItem {
  id: string
  label: string
  command?: AppCommandDetail
  icon?: string
  shortcutId?: string
  disabled?: boolean
  danger?: boolean
  separatorBefore?: boolean
  children?: ContextMenuItem[]
}
```

### 5.4 渲染层组件

建议新增：

```text
src/components/common/ContextMenu.vue
```

职责：

- 接收菜单项。
- 根据鼠标位置展示。
- 自动防止超出窗口边界。
- 点击菜单项后关闭。
- 支持禁用态、危险态、分割线、二级菜单。

### 5.5 触发方式

各区域通过 `contextmenu` 事件触发：

```ts
event.preventDefault()
showContextMenu({
  scope,
  x: event.clientX,
  y: event.clientY,
  payload,
})
```

`payload` 可包含：

- 当前文件路径
- 当前标签 id
- 当前选区
- 当前链接 URL
- 当前图片 src
- 当前编辑模式

### 5.6 与 Electron 原生菜单的关系

第一版建议使用 Vue 自绘菜单。

原因：

- 更容易与现有主题一致。
- 更容易做上下文禁用态。
- 更容易复用前端 command。
- 不需要为每个菜单项绕 IPC 到主进程。

后续如果要接入系统原生菜单，可以再评估 Electron `Menu.buildFromTemplate()`。

## 6. 关键交互细节

### 6.1 菜单关闭规则

菜单应在以下情况关闭：

- 点击菜单项后
- 点击页面其他区域
- 按下 Escape
- 滚动编辑区
- 切换标签
- 窗口失焦

### 6.2 菜单定位

右键菜单应根据窗口边界自动调整位置：

- 靠右时向左展开。
- 靠底部时向上展开。
- 二级菜单同样要防止溢出。

### 6.3 禁用态说明

禁用项可以保留但不可点击。

适合保留禁用态的场景：

- 没有选中文本时“复制”禁用。
- 当前文件未保存时“复制文件路径”禁用。
- WYSIWYG 不支持某命令时禁用并展示 title。

适合直接隐藏的场景：

- 非图片区域不显示图片菜单。
- 非链接区域不显示链接菜单。
- 非代码块区域不显示代码块菜单。

## 7. 风险与注意事项

### 7.1 不要复制业务逻辑

右键菜单只应该派发命令，不应该自己实现保存、删除、格式化等业务。

### 7.2 不要绕过文件安全模型

文件树、图片、导出相关菜单必须继续走现有 IPC 权限模型。

尤其是：

- 在系统资源管理器中显示
- 替换图片
- 删除图片文件
- 复制路径

都要注意不要恢复 renderer 任意访问本地路径的问题。

### 7.3 WYSIWYG 和源码模式能力要分开

右键菜单必须根据 `settingsStore.editorMode` 判断能力。

不要因为源码模式支持某命令，就在 WYSIWYG 中直接展示可点击入口。

### 7.4 右键菜单不替代顶部菜单

顶部菜单负责完整功能导航。

右键菜单负责当前场景的高频操作。

两者应复用命令，但产品定位不同。

## 8. 推荐第一批实现清单

建议第一批只做这些，范围可控：

1. 新增统一 `ContextMenu.vue`。
2. 新增 `contextMenuRegistry.ts`。
3. 编辑区接入基础文本菜单。
4. 文件树接入文件 / 文件夹 / 空白区域菜单。
5. 标签栏接入保存 / 关闭 / 关闭其他 / 复制路径。
6. 删除操作接入现有确认和错误提示。

暂不做：

- 图片替换
- 删除图片文件
- 插件菜单
- 用户自定义菜单
- 复杂二级菜单

## 9. 验收标准

完成第一阶段后，应满足：

1. 编辑区右键能显示基础文本操作。
2. 有选中文本和无选中文本时菜单状态不同。
3. 文件树文件、文件夹、空白区域右键菜单不同。
4. 标签页右键可关闭当前标签和关闭其他标签。
5. 未保存标签关闭时仍触发保存确认。
6. 文件删除失败不会关闭对应标签。
7. 所有右键菜单项复用现有命令或 store 方法。
8. 右键菜单样式适配当前主题。
9. 按 Escape 和点击空白处可关闭菜单。
10. `npm run type-check` 通过。

## 10. 结论

右键菜单是当前项目值得补齐的桌面端体验能力。

建议采用“统一菜单注册 + 上下文识别 + 复用现有命令”的方式分阶段实现。第一阶段先补齐编辑区、文件树、标签页的基础高频操作；第二阶段再加入 Markdown 格式化；第三阶段补齐预览、链接、代码块和图片菜单。

这样既能快速提升可用性，也能避免右键菜单变成分散、重复、难维护的另一套业务入口。
