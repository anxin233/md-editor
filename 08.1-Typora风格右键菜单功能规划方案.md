# 08.1-Typora风格右键菜单功能规划方案

关联文档：`08-右键菜单功能规划与分阶段落地方案.md`

## 1. 重新定位

上一版 `08` 更偏“桌面应用基础右键菜单”，覆盖编辑区、文件树、标签页等通用场景。

如果目标是接近 Typora，右键菜单的重心需要调整：它不应该只是“剪切 / 复制 / 粘贴 / 文件操作”的集合，而应该围绕 Markdown 写作上下文提供即时编辑能力。

Typora 风格右键菜单的核心特征是：

1. 更关注当前光标所在的 Markdown 语义块。
2. 更强调段落、标题、列表、引用、代码块、表格、图片、链接等结构化操作。
3. 菜单随上下文变化，而不是固定一套。
4. 编辑操作优先，文件管理操作次之。
5. 右键菜单应像“就地编辑工具箱”，而不是顶部菜单的简单复制。

## 2. 产品目标

目标体验：

用户在编辑正文时，不需要频繁移动到顶部菜单，也不需要记住所有 Markdown 语法。右键当前文本、段落、图片、链接、表格或代码块，就能完成最自然的下一步操作。

具体目标：

1. 让源码模式和 WYSIWYG 模式都拥有接近 Typora 的就地编辑体验。
2. 让不同 Markdown 元素拥有专属菜单。
3. 让右键菜单成为格式化、插入、修改结构的高频入口。
4. 保持菜单短小清晰，避免一次展示过多低频项。
5. 所有菜单项复用已有命令体系，避免行为分叉。

## 3. 右键菜单场景模型

Typora 风格右键菜单建议按“当前命中的对象”来决定菜单。

优先级从高到低：

1. 表格单元格
2. 图片
3. 链接
4. 代码块
5. 数学公式块
6. 任务列表项
7. 普通列表项
8. 引用块
9. 标题
10. 普通段落
11. 编辑区空白处
12. 文件树 / 标签页

当多个上下文重叠时，优先使用更具体的菜单。例如图片被包在链接中时，可以显示图片菜单，同时提供“打开链接 / 复制链接地址”。

## 4. 编辑区通用菜单

适用场景：

- 普通段落
- 标题
- 列表项
- 引用块
- 没有命中特殊元素的编辑区

### 4.1 有选中文本

菜单建议：

1. 剪切
2. 复制
3. 粘贴
4. 删除
5. 加粗
6. 斜体
7. 删除线
8. 行内代码
9. 超链接
10. 清除样式

二级菜单：

```text
段落
  段落
  标题 1
  标题 2
  标题 3
  标题 4
  标题 5
  标题 6
```

```text
列表
  无序列表
  有序列表
  任务列表
```

### 4.2 无选中文本

菜单建议：

1. 粘贴
2. 全选
3. 插入
4. 段落
5. 列表
6. 引用
7. 代码块
8. 公式块

插入二级菜单：

```text
插入
  图片
  链接
  表格
  分割线
  TOC 标记
  YAML Front Matter
```

## 5. 段落与标题菜单

当光标位于普通段落或标题行时，应提供块级结构转换。

建议菜单：

1. 段落
2. 标题 1
3. 标题 2
4. 标题 3
5. 标题 4
6. 标题 5
7. 标题 6
8. 提升标题级别
9. 降低标题级别

如果当前已经是标题：

- 当前标题级别显示选中态。
- 提供“转换为段落”。

如果当前是普通段落：

- “段落”显示选中态。

## 6. 列表菜单

### 6.1 普通列表项

命中无序列表或有序列表时：

1. 退出列表
2. 转为无序列表
3. 转为有序列表
4. 转为任务列表
5. 增加缩进
6. 减少缩进

### 6.2 任务列表项

命中任务列表时：

1. 标记为完成 / 标记为未完成
2. 转为普通列表
3. 转为有序列表
4. 增加缩进
5. 减少缩进

任务列表菜单应比普通列表更靠近 Typora 的写作体验，用户右键任务即可切换完成状态。

## 7. 引用块菜单

命中引用块时：

1. 退出引用
2. 转为普通段落
3. 转为标题
4. 转为无序列表
5. 转为有序列表

引用块内如果有选中文本，仍应保留基础格式化：

- 加粗
- 斜体
- 删除线
- 行内代码

## 8. 链接菜单

命中链接文本时：

1. 打开链接
2. 编辑链接
3. 复制链接地址
4. 取消链接
5. 复制

如果链接是本地文件路径：

1. 打开文件
2. 在系统资源管理器中显示

安全注意：

- 外部链接仍通过主进程 `openExternal` 打开。
- 只允许 `https://` 或经过明确允许的协议。
- 不要让 renderer 任意打开本地路径。

## 9. 图片菜单

命中图片时：

1. 查看原图
2. 复制图片
3. 复制图片路径
4. 编辑图片替代文本
5. 替换图片
6. 删除图片引用
7. 在系统资源管理器中显示

可选二级菜单：

```text
图片大小
  原始大小
  适应页面
  50%
  75%
  100%
```

第一阶段不建议实现图片缩放，因为 Markdown 标准语法本身不支持统一图片尺寸，需要额外 HTML 或扩展语法。

建议优先实现：

- 复制图片路径
- 编辑替代文本
- 删除图片引用
- 查看原图

删除策略：

- “删除图片引用”只删除 Markdown 中的图片语法。
- 不默认删除本地图片文件。
- 如果后续提供“删除本地图片文件”，必须二次确认。

## 10. 表格菜单

表格是 Typora 右键菜单中最重要的场景之一。

命中表格单元格时：

1. 插入行到上方
2. 插入行到下方
3. 插入列到左侧
4. 插入列到右侧
5. 删除当前行
6. 删除当前列
7. 删除表格

二级菜单：

```text
对齐方式
  左对齐
  居中
  右对齐
```

```text
表格
  插入行到上方
  插入行到下方
  插入列到左侧
  插入列到右侧
  删除行
  删除列
  删除表格
```

当前项目已有 `TableEditor.vue`，但右键菜单里的表格操作不应只是“打开表格编辑器”。长期应支持对当前 Markdown 表格的就地结构修改。

第一阶段可以先提供：

- 打开表格编辑器
- 删除表格

第二阶段再做行列级操作。

## 16. 分阶段落地计划

### 第一阶段：Typora 风格基础编辑菜单

目标：把当前已做的基础菜单升级为写作导向。

范围：

1. 编辑区菜单增加：
   - 加粗
   - 斜体
   - 删除线
   - 行内代码
   - 超链接
   - 段落二级菜单
   - 列表二级菜单
   - 插入二级菜单
2. 根据是否选中文本启用 / 禁用相关项。
3. WYSIWYG 不支持的命令禁用或隐藏。
4. 保留文件树和标签页当前第一阶段能力。

验收标准：

- 选中文本后右键可做基础 Markdown 格式化。
- 光标在段落内右键可转换标题 / 段落 / 列表。
- 插入二级菜单可触发表格、代码块、公式块等已有命令。
- 源码模式和 WYSIWYG 模式行为不冲突。

### 第二阶段：链接、图片、代码块专属菜单

目标：让常见 Markdown 元素拥有专属菜单。

范围：

1. 链接：
   - 打开链接
   - 编辑链接
   - 复制链接地址
   - 取消链接
2. 图片：
   - 查看原图
   - 复制图片路径
   - 编辑替代文本
   - 删除图片引用
3. 代码块：
   - 复制代码
   - 设置语言
   - 删除代码块

验收标准：

- 右键链接时不显示普通段落菜单作为主菜单。
- 右键图片时显示图片专属菜单。
- 右键代码块时可复制代码内容。

### 第三阶段：表格菜单

目标：补齐 Typora 风格表格编辑体验。

范围：

1. 插入行到上方 / 下方。
2. 插入列到左侧 / 右侧。
3. 删除行。
4. 删除列。
5. 删除表格。
6. 设置列对齐方式。

验收标准：

- Markdown 表格可通过右键进行结构调整。
- 表格操作不会破坏原有表格内容。
- 源码模式和 WYSIWYG 模式行为一致或有明确提示。

### 第四阶段：公式、任务列表和高级块操作

范围：

1. 公式块编辑。
2. 行内公式转公式块。
3. 任务列表完成状态切换。
4. 列表缩进 / 反缩进。
5. 引用块退出 / 转换。

验收标准：

- 任务列表右键可切换完成状态。
- 列表右键可调整缩进。
- 公式右键可复制源码。

## 17. 技术实现建议

### 17.1 从“区域菜单”升级为“语义菜单”

当前实现偏区域：

- CodeMirror 菜单
- WYSIWYG 菜单
- 文件树菜单
- 标签页菜单

Typora 风格需要进一步识别语义：

- 当前是否在链接上
- 当前是否在图片上
- 当前是否在表格中
- 当前是否在代码块中
- 当前是否在列表中
- 当前是否在标题中

### 17.2 建议新增上下文识别层

建议扩展：

```text
src/utils/contextMenuRegistry.ts
```

新增：

```ts
type EditorContextKind =
  | 'selection'
  | 'paragraph'
  | 'heading'
  | 'list'
  | 'task-list'
  | 'blockquote'
  | 'link'
  | 'image'
  | 'table'
  | 'code-block'
  | 'math'
  | 'empty'
```

并提供：

```ts
detectCodeMirrorContext(view, event): EditorContext
detectWysiwygContext(root, event): EditorContext
buildEditorContextMenu(context): ContextMenuEntry[]
```

### 17.3 命令仍应复用现有体系

菜单项不直接改文档，优先派发已有命令：

- `format:bold`
- `format:italic`
- `format:hyperlink`
- `paragraph:heading`
- `format:table`
- `format:code-block`
- `format:math-block`

如果某些命令只有源码模式支持，则 WYSIWYG 中应禁用或转换为对应 Milkdown 命令。

### 17.4 二级菜单支持

Typora 风格菜单需要二级菜单。

建议扩展 `ContextMenuEntry`：

```ts
interface ContextMenuEntry {
  id: string
  label: string
  disabled?: boolean
  danger?: boolean
  separatorBefore?: boolean
  action?: () => void | Promise<void>
  children?: ContextMenuEntry[]
}
```

第一阶段可以先用扁平菜单模拟，第二阶段补真正二级菜单。

## 18. 需要避免的问题

1. 不要把所有顶部菜单都塞进右键菜单。
2. 不要在 WYSIWYG 中显示无法执行的源码命令。
3. 不要通过不可靠的字符串查找修改 WYSIWYG 当前选区。
4. 不要让图片和本地文件操作绕过现有文件安全边界。
5. 不要让右键菜单成为另一套分散业务逻辑。

## 19. 推荐下一步

当前第一阶段基础菜单已经落地，下一步建议不是继续扩文件树，而是把编辑区升级为 Typora 风格：

1. 给 `ContextMenuEntry` 增加 `children` 支持。
2. 给编辑区增加上下文识别。
3. 先实现段落、标题、列表、插入二级菜单。
4. 再做链接、图片、代码块专属菜单。
5. 最后做表格行列级菜单。

## 20. 结论

如果目标是 Typora 风格，右键菜单应以“Markdown 语义元素”为中心，而不是只以“编辑区 / 文件树 / 标签页”为中心。

建议保留当前第一阶段基础右键能力，并在其上继续演进：

- 从基础文本菜单升级为 Markdown 格式菜单。
- 从区域识别升级为语义上下文识别。
- 从单层菜单升级为轻量二级菜单。
- 优先补链接、图片、代码块和表格这些 Typora 用户最熟悉的右键能力。

这样才能让右键菜单真正接近 Typora 的写作体验，而不是普通桌面应用的通用右键菜单。
