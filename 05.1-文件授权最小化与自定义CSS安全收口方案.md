# 05.1-文件授权最小化与自定义CSS安全收口方案

关联方案：`05-文件访问授权边界与导出资源安全方案.md`  
关联问题：

1. 自定义 CSS 持久化接口可被 renderer 伪造路径，形成任意文本文件读取。
2. 打开单个文件 / 保存文件 / 打开最近文件时授权整个父目录，导致同目录下其他文件获得过宽读写权限。

## 1. 当前验证结论

`05` 的主体改造方向正确：

- 已删除 `file.authorize`
- 已删除 `file:grantAccess`
- 已删除通用 `readBinary` 暴露
- 导出图片改为主进程专用 `file:readDocumentImage`
- 导出图片已限制相对路径、扩展名、大小和图片魔数

但仍有两个安全边界没有真正收口：

1. `settings:saveCustomCssPath(cssPath)` 仍允许 renderer 写入任意 CSS 路径记录。
2. `grantParentOfFile(filePath)` 实际把文件父目录放入 `directoryRoots`，而 `directoryRoots` 同时允许读和写。

这两个问题会削弱 `05` 的核心目标：renderer 只能请求操作，不能扩大权限。

## 2. 问题一：自定义 CSS 任意读

### 2.1 当前风险

当前设计中，preload 暴露了：

```ts
settings.saveCustomCssPath(cssPath)
settings.loadPersistedCustomCss()
settings.clearCustomCssPath()
```

主进程 `settings:saveCustomCssPath` 接收 renderer 传入的路径并写入：

```text
userData/custom-css-path.json
```

之后 `settings:loadPersistedCustomCss` 会读取该路径并返回文件内容。

风险链路：

```ts
await window.electronAPI.settings.saveCustomCssPath('C:/Users/ASUS/Desktop/secret.txt')
const result = await window.electronAPI.settings.loadPersistedCustomCss()
```

如果没有强校验，renderer 就可以把任意文本文件伪装成“已持久化 CSS”，再让主进程读取并返回内容。

这等价于重新引入了一个只读版 `file.authorize + file.read`。

### 2.2 目标

自定义 CSS 的安全目标：

1. renderer 不能指定任意路径让主进程持久化。
2. 持久化 CSS 路径只能来自 `dialog:openCssFile` 的真实选择结果。
3. 主进程只允许读取 `.css` 文件。
4. 启动恢复时只读取主进程曾经记录过的 CSS 文件。
5. 不再信任 renderer localStorage 中的 CSS 路径。

### 2.3 推荐方案 A：CSS 选择和持久化合并到主进程

推荐把 CSS 的“选择、授权、读取、持久化”合并为一个主进程可信流程。

新增 IPC：

```ts
settings:pickCustomCss
```

返回：

```ts
type CustomCssPayload = {
  path: string
  content: string
}
```

主进程流程：

1. 打开系统 CSS 文件选择框。
2. 用户选择文件后，主进程校验扩展名为 `.css`。
3. 主进程调用 `grantSingleFileRead(cssPath)`。
4. 主进程读取 CSS 内容。
5. 主进程把 CSS 路径写入 `userData/custom-css-path.json`。
6. 返回 `{ path, content }` 给 renderer。

renderer 不再调用：

```ts
dialog.openCssFile()
file.read(cssPath)
settings.saveCustomCssPath(cssPath)
```

而是调用：

```ts
settings.pickCustomCss()
```

### 2.4 preload API 调整

删除：

```ts
settings.saveCustomCssPath(cssPath)
```

保留：

```ts
settings.loadPersistedCustomCss()
settings.clearCustomCssPath()
```

新增：

```ts
settings.pickCustomCss(): Promise<{ path: string; content: string } | null>
```

可选删除：

```ts
dialog.openCssFile()
```

如果保留 `dialog.openCssFile()`，它只返回路径，不应顺带成为 CSS 持久化入口。长期建议把 CSS 选择放进 `settings` namespace，避免调用链分散。

### 2.5 settings.ipc.ts 改造

建议主进程只提供这三个接口：

```ts
settings:pickCustomCss
settings:loadPersistedCustomCss
settings:clearCustomCssPath
```

不要再提供：

```ts
settings:saveCustomCssPath
```

`settings:pickCustomCss` 示例流程：

```ts
ipcMain.handle('settings:pickCustomCss', async () => {
  const win = BrowserWindow.getFocusedWindow()
  if (!win) return null

  const result = await dialog.showOpenDialog(win, {
    properties: ['openFile'],
    filters: [
      { name: 'CSS', extensions: ['css'] },
    ],
  })

  if (result.canceled || result.filePaths.length === 0) return null

  const cssPath = resolve(result.filePaths[0])
  if (extname(cssPath).toLowerCase() !== '.css') return null

  grantSingleFileRead(cssPath)
  const content = await readFile(cssPath, 'utf-8')
  await saveCustomCssPath(cssPath)

  return { path: cssPath, content }
})
```

`settings:loadPersistedCustomCss` 仍然可以读取主进程记录的路径，但要补充校验：

1. JSON 中的 path 必须是字符串。
2. `resolve(path)` 后必须存在。
3. 扩展名必须是 `.css`。
4. 文件大小建议限制，例如 1 MB。
5. 读取前重新 `grantSingleFileRead(abs)`。

### 2.6 settings.ts 调整

renderer store 中：

```ts
pickAndLoadCustomCss()
```

应改为：

```ts
async function pickAndLoadCustomCss() {
  const picked = await window.electronAPI?.settings.pickCustomCss()
  if (!picked) return false
  applyCustomCssContent(picked.content)
  customCssPath.value = picked.path
  localStorage.removeItem('md-editor-custom-css')
  return true
}
```

`loadCustomCss(cssPath)` 如果继续保留，应只作为内部函数使用，不能接受任意外部路径并触发主进程持久化。

建议改名为：

```ts
applyPickedCustomCss(payload)
```

避免误用。

### 2.7 验收标准

完成后应满足：

1. preload 中不存在 `settings.saveCustomCssPath`。
2. renderer 不能传入任意路径让主进程持久化。
3. CSS 路径只能由 `settings:pickCustomCss` 的系统对话框产生。
4. `settings:loadPersistedCustomCss` 只读取主进程此前保存的 `.css` 路径。
5. localStorage 中的旧 `md-editor-custom-css` 只做清理，不作为授权来源。
6. 尝试从 renderer 伪造 `C:/xxx/secret.txt` 不应有任何公开 API 能保存并读取。

## 3. 问题二：父目录完整授权过宽

### 3.1 当前风险

当前 `grantParentOfFile(filePath)`：

```ts
export function grantParentOfFile(absFilePath: string): void {
  grantDirectoryRoot(dirname(resolve(absFilePath)))
}
```

而 `directoryRoots` 同时被：

```ts
assertPathReadable()
assertPathWritable()
```

使用。

这会导致：

1. 用户只打开 `D:/docs/a.md`。
2. 主进程授权了 `D:/docs`。
3. renderer 可以尝试读取、写入、删除、重命名 `D:/docs` 下其他文件。

这不符合最小权限原则。

### 3.2 目标

文件权限应区分：

1. 单文件读。
2. 单文件写。
3. 单文件 watch。
4. 文档目录资源读。
5. 文档目录图片保存。
6. 工作区目录完整管理。

只有“打开文件夹”才应该授予目录级管理能力。

### 3.3 推荐权限模型

把当前的：

```ts
directoryRoots: Set<string>
explicitReadFiles: Set<string>
```

改为结构化权限：

```ts
type FileOperation =
  | 'readText'
  | 'writeText'
  | 'watch'
  | 'readDir'
  | 'create'
  | 'rename'
  | 'delete'
  | 'saveImage'
  | 'readDocumentImage'

interface FilePermission {
  root: string
  kind: 'file' | 'directory'
  operations: Set<FileOperation>
  source:
    | 'open-file-dialog'
    | 'save-file-dialog'
    | 'open-folder-dialog'
    | 'recent-open'
    | 'custom-css-dialog'
}
```

所有校验统一走：

```ts
assertFileAccess(targetPath, operation)
```

### 3.4 授权函数设计

建议提供这些主进程内部函数：

```ts
grantOpenedMarkdownFile(filePath: string): void
grantSavedMarkdownFile(filePath: string): void
grantWorkspaceDirectory(dirPath: string): void
grantCustomCssFile(filePath: string): void
```

不要再提供语义模糊的：

```ts
grantParentOfFile(filePath)
grantDirectoryRoot(dirPath)
```

如果为了兼容短期保留，也要明确命名：

```ts
grantWorkspaceDirectoryRoot(dirPath)
```

### 3.5 打开单个文件的权限

用户通过 `dialog:openFile` 选择文件后，只授予：

```ts
{
  kind: 'file',
  root: filePath,
  operations: ['readText', 'writeText', 'watch']
}
```

同时，为了支持该文档相对图片导出和粘贴图片，可以额外授予文档目录的窄权限：

```ts
{
  kind: 'directory',
  root: dirname(filePath),
  operations: ['readDocumentImage', 'saveImage']
}
```

注意：这不是完整 `readDir/write/delete/rename` 权限。

### 3.6 保存文件 / 另存为权限

用户通过 `dialog:saveFile` 选择保存目标后，授予：

```ts
{
  kind: 'file',
  root: filePath,
  operations: ['readText', 'writeText', 'watch']
}
```

并授予父目录：

```ts
{
  kind: 'directory',
  root: dirname(filePath),
  operations: ['readDocumentImage', 'saveImage']
}
```

不授予父目录的 `delete`、`rename`、`create`、`readDir`。

### 3.7 打开工作区权限

只有 `dialog:openFolder` 可以授予目录级完整管理权限：

```ts
{
  kind: 'directory',
  root: folderPath,
  operations: [
    'readText',
    'writeText',
    'watch',
    'readDir',
    'create',
    'rename',
    'delete',
    'saveImage',
    'readDocumentImage',
  ]
}
```

这与用户选择“打开文件夹”的意图一致。

### 3.8 打开最近文件权限

`recent:open` 应先验证：

1. 请求路径存在于主进程 recent 列表。
2. 文件存在。
3. 文件扩展名符合 Markdown 文档类型，至少建议限制为 `.md/.markdown/.mdown/.mkd`。

然后授予与打开单个文件相同的权限：

```ts
grantOpenedMarkdownFile(filePath)
```

不要调用目录级完整授权。

### 3.9 assert 函数拆分

现有：

```ts
assertPathReadable(path)
assertPathWritable(path)
```

容易把“能读文本文件”和“能读目录资源”混在一起。

建议改成：

```ts
assertCanReadText(path)
assertCanWriteText(path)
assertCanReadDir(path)
assertCanCreate(path)
assertCanRename(oldPath, newPath)
assertCanDelete(path)
assertCanWatch(path)
assertCanSaveImage(dirPath)
assertCanReadDocumentImage(path)
```

或者保留一个通用函数：

```ts
assertFileAccess(path, 'readText')
```

但调用点必须传入明确 operation。

### 3.10 IPC 对应关系

建议每个 IPC 明确对应 operation：

```text
file:read              -> readText
file:write             -> writeText
file:readDir           -> readDir
file:stat              -> readText 或 readDir，按目标类型判断
file:exists            -> 不返回未授权路径存在性；未授权统一 false
file:createFile        -> create
file:createDir         -> create
file:rename            -> rename
file:delete            -> delete
file:saveImage         -> saveImage
file:watch             -> watch
file:readDocumentImage -> readDocumentImage
```

其中 `file:readDocumentImage` 应只接受文档源路径和图片相对路径，不应允许 renderer 直接传最终绝对图片路径。

## 4. 图片保存补充要求

虽然本 `05.1` 重点是两个问题，但建议顺手补齐 `file:saveImage` 的魔数校验。

当前粘贴图片保存只校验：

- 文件名扩展名
- 文件大小

建议复用导出图片中的魔数判断：

```ts
assertImageSignature(buffer, ext)
```

保存前确认：

1. `.png` 必须是 PNG 魔数。
2. `.jpg/.jpeg` 必须是 JPEG 魔数。
3. `.gif` 必须是 GIF 魔数。
4. `.webp` 必须是 WebP 魔数。

这样 renderer 不能通过传入 `fake.png` 写入任意二进制内容。

## 5. 推荐实现顺序

### 第一阶段：修自定义 CSS 任意读

1. 删除 preload 中的 `settings.saveCustomCssPath`。
2. 删除 `settings:saveCustomCssPath` IPC。
3. 新增 `settings:pickCustomCss`。
4. CSS 选择、读取、路径保存全部放在主进程内完成。
5. `settings.ts` 改为调用 `settings.pickCustomCss()`。
6. `settings:loadPersistedCustomCss` 增加 `.css`、存在性、大小限制。

这一阶段完成后，renderer 不再能伪造 CSS 路径。

### 第二阶段：重构文件权限模型

1. 用 `FilePermission[]` 替代 `directoryRoots + explicitReadFiles`。
2. 引入 operation 级别校验。
3. 删除或停用 `grantParentOfFile()`。
4. 新增：
   - `grantOpenedMarkdownFile`
   - `grantSavedMarkdownFile`
   - `grantWorkspaceDirectory`
   - `grantCustomCssFile`
5. 修改 dialog 和 recent 调用点。

### 第三阶段：收紧 IPC operation

1. `file:read` 使用 `readText`。
2. `file:write` 使用 `writeText`。
3. `file:readDir` 只能使用 `readDir`。
4. `file:delete` 只能使用 `delete`。
5. `file:rename` 只能使用 `rename`。
6. `file:saveImage` 只能使用 `saveImage`。
7. `file:readDocumentImage` 只能使用 `readDocumentImage`。

### 第四阶段：补图片保存魔数校验

1. 抽出 `sniffImageMime()`。
2. 导出图片和粘贴图片保存共用校验逻辑。
3. 保存接口中校验扩展名、大小、魔数一致。

### 第五阶段：验证

执行：

```bash
npm run type-check
npx vite build
```

并补充手动验证：

1. 打开单个 Markdown 文件后，不能读取同目录其他未打开文件。
2. 打开单个 Markdown 文件后，不能删除同目录其他文件。
3. 打开工作区后，工作区内文件树操作正常。
4. 自定义 CSS 可正常选择、应用、重启恢复。
5. renderer 无法通过公开 API 保存任意 CSS 路径。
6. 导出同目录相对图片正常内联。
7. 导出绝对路径、`file:`、`../` 图片不会被内联。
8. 粘贴伪装扩展名的非图片文件应失败。

## 6. 验收标准

完成本方案后，应满足：

1. preload 不再暴露 `settings.saveCustomCssPath`。
2. 主进程不再注册 `settings:saveCustomCssPath`。
3. 自定义 CSS 路径只能由主进程 CSS 选择对话框产生。
4. `loadPersistedCustomCss` 只能读取主进程持久化过的 `.css` 文件。
5. 不存在 renderer 传入任意路径并让主进程读取返回的链路。
6. 打开单个文件不会授权整个父目录的完整读写。
7. 打开最近文件不会授权整个父目录的完整读写。
8. 保存 / 另存为不会授权父目录的完整读写。
9. 只有打开工作区才授予目录级文件管理能力。
10. 粘贴图片保存校验扩展名、大小和魔数。
11. `npm run type-check` 通过。
12. `npx vite build` 通过。

## 7. 不建议的修法

不要只做以下修补：

1. 在 renderer 里判断 CSS 路径后缀是不是 `.css`。
2. 保留 `saveCustomCssPath(cssPath)`，只在主进程里加简单字符串过滤。
3. 继续使用 `grantParentOfFile()`，但期望调用方“不要乱用”。
4. 用两个 Set 分别表示可读、可写目录，但不区分 `readDir/delete/rename/saveImage/readDocumentImage`。
5. 对打开单文件授予父目录写权限，只为了粘贴图片方便。

这些做法仍然没有建立清晰的最小权限模型。

## 8. 结论

`05` 已经移除了最危险的公开 `file.authorize`，但 `05.1` 必须继续收紧两个剩余入口：

1. 自定义 CSS 的路径持久化必须完全由主进程可信对话框控制。
2. 文件授权必须从“目录根可读写”升级为“操作级最小权限”。

完成后，文件安全模型才真正符合目标：renderer 不能授予权限，也不能通过其他 IPC 间接扩大权限；只有用户通过系统对话框表达过的操作意图，才会在主进程中转化为精确权限。
